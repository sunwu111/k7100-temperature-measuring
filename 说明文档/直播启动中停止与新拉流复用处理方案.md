# 直播启动中停止与新拉流复用处理方案

## 一、背景

当前唤醒模式下和正常模式下，红外或可见光启动直播过程中，如果后台或用户发起停止直播，设备侧还没有完整的“启动中停止保护”和“新拉流复用启动过程”处理。

现在需要补充的不是简单延长启动超时时间，而是解决下面这个状态交错问题：

| 场景 | 当前风险 |
| --- | --- |
| 启动直播还没有完成，后台发来停止直播 | 可能强制打断正在打开的设备，导致启动流程和停止流程交错 |
| 启动中停止已经响应，后续用户又发起新的拉流 | 如果没有复用逻辑，可能重复打开设备，或者旧停止把新拉流停掉 |
| 用户手动停止后，后台不再补发停止指令 | 如果设备内部没有记录停止意图，启动成功后可能没人再停 |

## 二、处理目标

| 目标 | 说明 |
| --- | --- |
| 停止命令必须立即响应 | 设备收到停止直播后，要立即向后台返回停止响应，不能让后台一直等待 |
| 启动流程不被强制打断 | 如果设备仍在上电、打开机芯、初始化编码器或准备出流，不立即硬关闭 |
| 停止意图不能丢失 | 启动中收到停止后，要记录“延迟停止请求” |
| 新拉流可以复用启动过程 | 如果启动过程中又收到新的拉流请求，取消旧的延迟停止，继续服务新的会话 |
| 没有新拉流时自动停止 | 如果后续没有新的拉流请求，启动完成或到达安全点后自动执行停止 |

## 三、适用范围

| 通道 | 设备 | 是否适用 |
| --- | --- | --- |
| 通道一 | 可见光机芯 | 适用 |
| 通道二 | 红外机芯 | 适用 |
| MIPI | MIPI 设备 | 本方案暂不处理，除非后续确认 MIPI 拉流也存在相同的启动中停止交错问题 |

本方案同时适用于唤醒模式和正常模式。

唤醒模式下，启动前可能涉及云台、可见光、红外上电和保活判断。

正常模式下，设备通常已经处于供电状态，但直播启动过程中仍然可能存在设备打开、编码器初始化、出流准备等阶段，所以也需要同样的停止保护。

## 四、核心状态

建议为每个直播通道维护独立状态，不要红外和可见光共用同一个启动状态。

| 状态 | 含义 |
| --- | --- |
| 未启动 | 当前没有直播任务 |
| 启动中 | 已收到开始直播，正在上电、打开设备或准备出流 |
| 已出流 | 直播已经成功 |
| 停止中 | 已收到停止，正在执行停止和资源释放 |

同时需要记录几个关键标志：

| 标志 | 含义 |
| --- | --- |
| 延迟停止请求 | 启动过程中收到过停止直播 |
| 当前会话标识 | 当前正在服务的拉流会话，例如 SSRC |
| 待复用会话标识 | 启动过程中又收到的新拉流会话 |
| 启动序号 | 每次开始直播递增，用来区分旧启动和新启动 |

## 五、红外启动中的停止处理

红外启动过程中收到停止直播时，分成两层处理。

| 层级 | 处理方式 |
| --- | --- |
| 命令响应层 | 立即向后台返回停止响应 |
| 设备执行层 | 不强制中断当前启动流程，只记录延迟停止请求 |

推荐流程：

| 步骤 | 处理 |
| --- | --- |
| 1 | 红外开始直播，进入启动中状态 |
| 2 | 启动过程中收到停止直播 |
| 3 | 立即返回停止响应 |
| 4 | 记录延迟停止请求 |
| 5 | 红外继续完成当前启动流程，直到成功、失败、超时或到达安全点 |
| 6 | 如果期间没有新的拉流请求，启动收口后执行停止 |
| 7 | 如果期间收到新的拉流请求，取消旧的延迟停止，复用当前启动过程 |

这里的关键点是：停止命令不能卡住不回，但也不能马上把正在打开的红外机芯强制关掉。

## 六、红外启动中收到新拉流

如果红外启动中已经收到过停止直播，但设备还没有真正执行停止，此时又收到新的红外拉流请求，应按下面方式处理。

| 条件 | 处理 |
| --- | --- |
| 新拉流到来时红外仍在启动中 | 不重新打开红外，复用当前启动过程 |
| 新拉流的 SSRC 和旧 SSRC 不同 | 更新当前会话标识为新 SSRC |
| 已经存在延迟停止请求 | 取消旧的延迟停止请求 |
| 红外启动成功 | 直接服务新的 SSRC |
| 红外启动失败或超时 | 返回新拉流失败，并清理状态 |

这样可以避免“旧停止命令把新拉流停掉”。

## 七、红外没有新拉流时如何停止

如果红外启动过程中收到停止直播，后续没有新的红外拉流请求，则设备不能一直保持直播。

推荐处理如下：

| 启动结果 | 后续处理 |
| --- | --- |
| 启动成功 | 立即执行正常停止直播流程 |
| 启动失败 | 清理启动状态，释放已打开资源 |
| 启动超时 | 清理启动状态，释放已打开资源 |
| 到达安全点 | 执行停止和资源释放 |

安全点可以理解为：当前阶段已经不会因为立即关闭导致状态错乱的位置，例如设备打开流程已经返回、编码器初始化已经结束、预览启动流程已经收口。

## 八、可见光启动中的停止处理

可见光处理原则和红外一致。

| 场景 | 处理 |
| --- | --- |
| 可见光启动中收到停止直播 | 立即响应停止命令 |
| 设备仍在打开中 | 不强制中断，记录延迟停止请求 |
| 后续没有新 SSRC 拉流 | 到安全点后执行停止 |
| 后续有新 SSRC 拉流 | 旧停止作废，继续服务新 SSRC |

可见光和红外的差异主要在设备链路上。

| 设备 | 启动链路 |
| --- | --- |
| 可见光 | 云台、可见光机芯、预置位、编码器、直播出流 |
| 红外 | 云台、可见光 ready、红外上电、USB 识别、红外机芯、编码器、直播出流 |

虽然链路不同，但“停止立即响应，执行层延迟收口，新拉流复用启动过程”的原则一致。

## 九、唤醒模式下的特殊处理

唤醒模式下，停止直播后不能只看直播状态，还要结合保活和拍照开启窗口。

| 场景 | 处理 |
| --- | --- |
| 启动中收到停止，后续没有新拉流 | 启动收口后停止直播资源 |
| 停止后仍在业务保活时间内 | 按保活策略决定是否继续给云台、可见光、红外供电 |
| 停止后处于定时拍照提前开启窗口 | 不应因为直播停止而破坏拍照窗口需要的供电 |
| 停止后不在保活时间，也不在拍照窗口 | 可以按唤醒模式策略下电 |

也就是说，直播停止只表示直播业务结束，不一定等于马上对设备下电。

## 十、正常模式下的特殊处理

正常模式下，设备通常不需要像唤醒模式一样频繁上下电。

| 场景 | 处理 |
| --- | --- |
| 启动中收到停止，后续没有新拉流 | 启动收口后停止直播资源 |
| 后续有新 SSRC 拉流 | 复用当前启动过程并服务新 SSRC |
| 直播停止完成 | 释放直播、编码器、会话相关资源 |
| 是否下电 | 按正常模式原有设备供电策略处理，不因为本方案额外改变 |

正常模式重点解决的是命令交错和会话错乱，不重点处理上下电。

## 十一、典型时序

### 1. 启动中停止，后续没有新拉流

| 时间 | 事件 | 设备处理 |
| --- | --- | --- |
| 0 秒 | 收到红外开始直播 | 进入启动中 |
| 5 秒 | 收到停止直播 | 立即响应停止，记录延迟停止 |
| 12 秒 | 红外启动流程收口 | 检查到延迟停止 |
| 13 秒 | 执行停止流程 | 释放直播资源 |

### 2. 启动中停止，后续有新拉流

| 时间 | 事件 | 设备处理 |
| --- | --- | --- |
| 0 秒 | 收到红外开始直播，SSRC 为 A | 进入启动中 |
| 5 秒 | 收到停止 A | 立即响应停止，记录延迟停止 |
| 7 秒 | 收到新的红外开始直播，SSRC 为 B | 取消延迟停止，复用当前启动 |
| 12 秒 | 红外启动成功 | 服务 SSRC B |

### 3. 可见光启动中停止，后续有新 SSRC

| 时间 | 事件 | 设备处理 |
| --- | --- | --- |
| 0 秒 | 收到可见光开始直播，SSRC 为 A | 云台和可见光开始启动 |
| 4 秒 | 收到停止 A | 立即响应停止，记录延迟停止 |
| 6 秒 | 收到新可见光开始直播，SSRC 为 B | 旧停止作废，继续启动 |
| 10 秒 | 可见光启动成功 | 服务 SSRC B |

## 十三、建议增加的日志

| 日志点 | 说明 |
| --- | --- |
| 收到开始直播 | 记录通道、SSRC、当前状态 |
| 进入启动中 | 记录启动序号 |
| 启动中收到停止 | 记录停止对应的 SSRC |
| 停止命令已响应 | 确认后台响应没有被阻塞 |
| 设置延迟停止 | 确认设备内部记录了停止意图 |
| 启动中收到新拉流 | 记录旧 SSRC、新 SSRC |
| 取消延迟停止 | 确认新拉流接管当前启动 |
| 启动流程收口 | 记录成功、失败或超时 |
| 执行延迟停止 | 确认没有新拉流时自动停止 |
| 服务新 SSRC | 确认复用启动过程成功 |

## 十四、总结

本方案的核心不是“不响应停止”，而是：

停止命令立即响应。

设备内部延迟收口。

新拉流复用当前启动过程。

没有新拉流时自动停止并释放资源。

这样可以同时解决后台等待、用户手动停止后没有第二次停止指令、旧停止影响新拉流、启动和停止流程交错这几个问题。

## 十五、建议修改代码

下面内容是建议修改方向，先放在文档中用于分析，不表示已经修改源码。

### 1. 在 MainActivity 中增加直播启动会话状态

建议在 `MainActivity` 中增加每个通道独立的直播启动状态，用来记录当前通道是否正在启动、是否收到过停止、当前 SSRC 是谁、新 SSRC 是否接管当前启动流程。

```java
private static class LiveStartSession {
    boolean starting;
    boolean delayStop;
    int channel;
    int streamType;
    int currentSsrc;
    int pendingSsrc;
    String server;
    int port;
    int network;
    long startSeq;
}

private final Map<Integer, LiveStartSession> liveStartSessions = new HashMap<>();
private long liveStartSeq = 0;

private synchronized LiveStartSession getLiveStartSession(int channel) {
    LiveStartSession session = liveStartSessions.get(channel);
    if (session == null) {
        session = new LiveStartSession();
        session.channel = channel;
        liveStartSessions.put(channel, session);
    }
    return session;
}
```

建议含义：

| 字段 | 作用 |
| --- | --- |
| starting | 当前通道是否处于直播启动中 |
| delayStop | 启动中是否收到过停止请求 |
| currentSsrc | 当前正在启动或服务的 SSRC |
| pendingSsrc | 启动中又收到的新 SSRC |
| startSeq | 启动序号，避免旧回调影响新任务 |

### 2. 增加启动中收到新拉流的处理函数

当同一个通道正在启动时，如果又收到新的拉流请求，不重新打开设备，而是更新会话信息，取消旧停止标志，复用当前启动过程。

```java
private synchronized boolean reuseStartingLiveSession(int channel, int streamType, int network, int ssrc, String server, int port) {
    LiveStartSession session = getLiveStartSession(channel);
    if (!session.starting) {
        return false;
    }

    Log.i(Log.TAG, "直播启动中收到新拉流，复用当前启动流程，channel=" + channel
            + ", oldSsrc=" + session.currentSsrc
            + ", newSsrc=" + ssrc);

    session.streamType = streamType;
    session.network = network;
    session.pendingSsrc = ssrc;
    session.currentSsrc = ssrc;
    session.server = server;
    session.port = port;
    session.delayStop = false;
    return true;
}
```

这个函数的目的：

| 场景 | 处理 |
| --- | --- |
| 旧拉流启动中，收到停止 | 先记录延迟停止 |
| 后续新 SSRC 拉流到来 | 取消延迟停止 |
| 当前启动还没完成 | 继续复用当前启动 |
| 启动成功 | 服务新的 SSRC |

### 3. 修改 startLiveVideo 开头逻辑

建议在 `startLiveVideo` 获取到 `dev` 后，先判断当前通道是否正在启动。

原来逻辑里，只有 `dev.isLiving()` 判断，无法覆盖“正在启动但还没出流”的状态。

建议增加如下逻辑：

```java
final Device dev = channels.get(String.valueOf(channel));
if (dev == null) return 3;

if (reuseStartingLiveSession(channel, streamType, network, ssrc, server, port)) {
    return 0;
}
```

然后在真正开始打开设备前，记录启动会话。

```java
final LiveStartSession session = getLiveStartSession(channel);
synchronized (this) {
    session.starting = true;
    session.delayStop = false;
    session.channel = channel;
    session.streamType = streamType;
    session.network = network;
    session.currentSsrc = ssrc;
    session.pendingSsrc = 0;
    session.server = server;
    session.port = port;
    session.startSeq = ++liveStartSeq;
}
final long startSeq = session.startSeq;
```

注意点：

| 注意点 | 说明 |
| --- | --- |
| 这段要放在真正 `dev.open()` 之前 | 保证启动过程中停止能识别到状态 |
| `startSeq` 要传进回调 | 避免旧启动回调修改新会话 |
| 不能只依赖 `dev.isLiving()` | 启动中还没有进入 LIVING 状态 |

### 4. 修改 openSucceed 回调

当前 `openSucceed()` 里直接使用方法入参 `ssrc` 调用 `dev.liveStart(streamType, ssrc)`。

建议改成：启动成功时先检查当前会话是否已经被新 SSRC 接管，如果被接管，就使用最新 SSRC。

```java
public void openSucceed() {
    LiveStartSession session = getLiveStartSession(channel);
    int liveSsrc;
    boolean needStopAfterOpen;

    synchronized (MainActivity.this) {
        if (session.startSeq != startSeq) {
            Log.i(Log.TAG, "忽略旧直播启动回调，channel=" + channel + ", startSeq=" + startSeq);
            return;
        }
        liveSsrc = session.currentSsrc;
        needStopAfterOpen = session.delayStop;
    }

    boolean ret = dev.liveStart(streamType, liveSsrc);
    showMsg("直播预览：" + (ret ? "成功" : "失败"));

    if (!ret) {
        finishLiveStartSession(channel, startSeq);
        finishTask(TaskManager.Task.Living.toString());
        if (dev.isCamera()) {
            powerOffMipiIfIdle("MIPI直播预览启动失败");
        }
        return;
    }

    synchronized (MainActivity.this) {
        session.starting = false;
    }

    if (needStopAfterOpen) {
        Log.i(Log.TAG, "直播启动成功后执行延迟停止，channel=" + channel + ", ssrc=" + liveSsrc);
        stopLiveVideo(channel, streamType, liveSsrc);
        return;
    }

    updateOnline("直播预览");
}
```

这里要注意：

| 情况 | 处理 |
| --- | --- |
| 启动中没有收到停止 | 正常 `liveStart` 并出流 |
| 启动中收到停止，后续没有新拉流 | 启动成功后立即执行停止 |
| 启动中收到停止，后续有新拉流 | `delayStop` 已经被取消，启动成功后服务新 SSRC |

### 5. 修改 openFailed 回调

启动失败时，要清理启动状态，避免后续一直认为通道还在启动。

```java
public void openFailed(int errcode) {
    if (!isSameLiveStartSession(channel, startSeq)) {
        Log.i(Log.TAG, "忽略旧直播启动失败回调，channel=" + channel + ", startSeq=" + startSeq);
        return;
    }

    finishLiveStartSession(channel, startSeq);

    showMsg("直播预览：失败");
    Device dev = channels.get(String.valueOf(channel));
    if (dev != null && dev.isDVR()) {
        if (dev.isUSB()) {
            sleepDevice(channel, "红外直播预览失败");
        } else {
            sleepDevice(channel, "可见光直播预览失败");
        }
    }

    finishTask(TaskManager.Task.Living.toString());
    if (dev != null && dev.isCamera()) {
        powerOffMipiIfIdle("MIPI直播预览打开失败");
    }
}
```

配套函数：

```java
private synchronized boolean isSameLiveStartSession(int channel, long startSeq) {
    LiveStartSession session = liveStartSessions.get(channel);
    return session != null && session.startSeq == startSeq;
}

private synchronized void finishLiveStartSession(int channel, long startSeq) {
    LiveStartSession session = liveStartSessions.get(channel);
    if (session == null || session.startSeq != startSeq) {
        return;
    }
    session.starting = false;
    session.delayStop = false;
    session.pendingSsrc = 0;
}
```

### 6. 修改 stopLiveVideo 逻辑

当前 `stopLiveVideo` 会直接执行：

```java
dev.liveStop();
dev.streamClient.close();
sleepDevice(channel, "红外停止直播");
```

建议在真正停止前，先判断是否处于启动中。

```java
public boolean stopLiveVideo(int channel, int streamType, int ssrc) {
    Device dev = channels.get(String.valueOf(channel));

    if (markDelayStopIfLiveStarting(channel, streamType, ssrc)) {
        Log.i(Log.TAG, "直播启动中收到停止，已响应停止命令并记录延迟停止，channel="
                + channel + ", ssrc=" + ssrc);
        return true;
    }

    try {
        if (dev != null && !dev.isRecording()) {
            dev.liveStop();

            if (dev.streamClient != null) {
                dev.streamClient.close();
                dev.streamClient = null;
            }

            if (dev.isDVR()) {
                if (dev.isUSB()) {
                    sleepDevice(channel, "红外停止直播");
                } else {
                    sleepDevice(channel, "可见光停止直播");
                }
            }
            if (dev.isCamera()) {
                powerOffMipiIfIdle("MIPI停止直播");
            }
        }
        releaseDecoder();
        finishTask(TaskManager.Task.Living.toString());
    } catch (Exception e) {
        Log.i(Log.TAG, "停止预览异常：" + e);
    }
    return true;
}
```

配套函数：

```java
private synchronized boolean markDelayStopIfLiveStarting(int channel, int streamType, int ssrc) {
    LiveStartSession session = liveStartSessions.get(channel);
    if (session == null || !session.starting) {
        return false;
    }

    if (ssrc != 0 && session.currentSsrc != 0 && ssrc != session.currentSsrc) {
        Log.i(Log.TAG, "启动中收到旧 SSRC 停止，忽略延迟停止，channel=" + channel
                + ", stopSsrc=" + ssrc
                + ", currentSsrc=" + session.currentSsrc);
        return true;
    }

    session.delayStop = true;
    return true;
}
```

这里返回 `true` 的含义是：停止命令已经被设备接收并响应，但设备执行层暂时不立即强制关闭，后面会在启动流程收口后处理。

### 7. 红外 GUIDEDev.open 不建议继续被 stop 直接 closeForce 打断

当前 `GUIDEDev.open()` 中有：

```java
if (closeForce) return false;
```

并且 `close()` 中会设置：

```java
closeForce = true;
```

这会导致启动中收到停止时，红外打开流程直接返回失败。建议后续改法是：直播启动中的停止不要立即调用 `dev.close()` 或触发 `closeForce`，只在安全点后执行真正关闭。

如果必须在设备类内部区分，可以增加一个“允许打断打开”的参数或标志。

建议方向：

```java
private volatile boolean closeForce = false;
private volatile boolean liveStartDelayStop = false;

public void requestLiveStartDelayStop() {
    liveStartDelayStop = true;
}

public boolean hasLiveStartDelayStop() {
    return liveStartDelayStop;
}

public void clearLiveStartDelayStop() {
    liveStartDelayStop = false;
}
```

然后在 `MainActivity.stopLiveVideo()` 中，启动中停止只设置延迟停止，不直接触发设备关闭。

### 8. 可见光 MyOnvifDevice.open 也需要同样保护

当前可见光 `MyOnvifDevice.open()` 中也有：

```java
} while (Math.abs(System.currentTimeMillis() - bootTime) / 1000 < timeoutSeconds && !closeForce);
```

并且 `close()` 中会设置：

```java
closeForce = true;
```

所以可见光启动中收到停止时，也可能出现“打开球机中被强制关闭”的情况。

处理原则和红外一致：

| 场景 | 建议 |
| --- | --- |
| 可见光启动中收到停止 | 命令立即响应，设备内部记录延迟停止 |
| 后续没有新 SSRC | 打开流程收口后执行停止 |
| 后续有新 SSRC | 取消旧停止，复用当前打开流程 |
| 不建议 | 在启动过程中直接 close 导致 `closeForce = true` |

### 9. 需要保留现有唤醒模式保活逻辑

这次修改只处理“直播启动中停止”和“新拉流复用”，不应破坏原来的唤醒模式保活。

也就是说，延迟停止最后真正执行停止时，仍然走现有逻辑：

```java
sleepDevice(channel, "红外停止直播");
sleepDevice(channel, "可见光停止直播");
powerOffMipiIfIdle("MIPI停止直播");
finishTask(TaskManager.Task.Living.toString());
```

是否下电仍由现有保活时间、拍照提前开启窗口、设备忙闲状态共同决定。

### 10. 建议修改顺序

| 顺序 | 修改内容 | 说明 |
| --- | --- | --- |
| 1 | 先在 MainActivity 增加 LiveStartSession | 只增加状态，不改设备类 |
| 2 | 修改 startLiveVideo | 支持启动中新 SSRC 复用 |
| 3 | 修改 stopLiveVideo | 启动中停止立即响应，但只记录延迟停止 |
| 4 | 修改 openSucceed/openFailed | 启动收口后根据状态决定继续出流还是停止 |
| 5 | 验证红外启动中停止 | 看是否还会出现启动/停止交错 |
| 6 | 验证可见光启动中停止 | 看新 SSRC 是否能接管旧启动 |
| 7 | 再决定是否需要改 GUIDEDev/MyOnvifDevice | 如果 MainActivity 层已经拦住停止，设备类可能暂时不用改 |

建议第一版优先只改 `MainActivity`，不要先动 `GUIDEDev` 和 `MyOnvifDevice` 的底层打开逻辑。这样风险更低，也更容易回退。
