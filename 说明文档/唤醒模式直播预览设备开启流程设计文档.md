# 唤醒模式直播预览设备开启流程设计文档

## 一、文档范围

本文整理唤醒模式下发起直播预览时，系统如何开启可见光 `HuanYuDevice` 和红外 `GUIDEDev`。

这里的直播预览主要对应主站发起的实时预览流程。本地预览流程与其大体一致，也会经过唤醒、打开设备、启动预览这几个核心步骤。

## 二、相关设备类型

`HuanYuDevice` 表示寰宇可见光机芯，属于 DVR/ONVIF 设备。它不直接实现自己的打开流程，而是继承 `MyOnvifDevice` 的打开逻辑。

`GUIDEDev` 表示高德红外机芯，属于 USB 红外设备。它需要红外负载和 USB 支持，同时在打开红外时还会检查可见光云台是否 ready。

在业务判断上，系统不是只按通道号区分可见光和红外，而是通过设备类型判断：

- 非 USB 的 DVR 设备按可见光处理。
- USB 红外设备按红外处理。

## 三、唤醒模式的基础原则

唤醒模式下，云台和红外默认允许下电，只有业务需要时才临时上电。

直播预览、录像回放、录像文件查询、拍照、短视频等业务入口都会刷新唤醒保活。直播预览也在这个范围内。

直播预览开始前，系统会先标记一次唤醒业务。标记后会做两件事：

- 根据设备类型上电对应负载。
- 重置唤醒模式保活倒计时。

当前保活时间为 2 分钟。保活到期后，系统会再次检查是否仍有正在执行的业务。如果还有直播、拍照、录像、回放或设备正在打开，则继续保活；如果没有业务，则进入唤醒待机并尝试下电。

## 四、直播预览通用流程

主站请求直播预览后，系统先根据通道号找到对应设备。

如果当前处于休眠模式，直接拒绝直播预览。

如果设备已经在直播，直接返回已在直播状态，避免重复开启。

如果可以继续执行，系统会获取 CPU 锁，并把当前任务标记为直播任务。

如果设备是 DVR 类设备，会根据设备类型执行唤醒：

- 可见光预览：开启云台负载。
- 红外预览：开启云台负载和红外负载。

如果目标设备是摄像头类设备，还会检查是否有正在录像的摄像头。录像优先级更高，如果正在录像，则不继续开启直播。若其他摄像头正在直播或录像，会先停止其他摄像头，避免平台同时打开多个摄像头导致冲突。

随后系统准备网络推流客户端。如果推流客户端未创建，会按主站下发的服务器地址、端口和网络类型创建连接。连接失败时，直播预览直接失败。

准备好 OSD 后，系统调用设备自身的打开流程。打开成功后，才会进入真正的 `liveStart`。

## 五、唤醒模式下开启 HuanYuDevice 的流程

`HuanYuDevice` 是可见光 ONVIF 设备，直播预览时按可见光负载处理。

在唤醒模式下发起可见光直播预览时，系统会先刷新唤醒保活，然后开启云台负载。红外负载不会因为可见光预览而单独开启。

云台负载开启后，系统进入 `HuanYuDevice` 的打开流程。由于 `HuanYuDevice` 继承自 `MyOnvifDevice`，实际打开逻辑是 ONVIF 设备打开逻辑。

打开时会先判断设备是否已经 ready。如果 ONVIF 已经 ready，会直接回调打开成功，不再重复等待。

如果还没有 ready，系统会进入轮询等待。轮询过程中每秒检查一次设备是否 ready，最长等待时间由 DVR 启动超时时间控制，当前为 120 秒。

设备 ready 后，如果还没有读取到设备型号，会通过 ONVIF 获取设备信息。直播预览传入的自检等待参数为 false，因此可见光直播预览不会额外等待 15 秒云台自检时间。

打开成功后，系统执行直播预览回调：

- 更新 OSD 状态文本。
- 同步设备时间。
- 调用 `liveStart` 开启 RTSP 直播。
- 直播成功后刷新在线状态，延长主站在线窗口。

`HuanYuDevice` 的直播启动逻辑是 ONVIF/RTSP 预览。它会根据码流创建 RTP/H264 处理对象，停止旧的 RTSP 预览客户端，创建新的 RTSP 客户端并启动拉流。启动成功后，设备状态进入直播中。

如果打开失败，系统会显示直播失败，并按可见光预览失败处理：关闭或休眠可见光设备，结束直播任务，然后尝试重启可见光负载。

## 六、唤醒模式下开启 GUIDEDev 的流程

`GUIDEDev` 是高德 USB 红外机芯，直播预览时按红外负载处理。

在唤醒模式下发起红外直播预览时，系统会先刷新唤醒保活，然后同时唤醒云台和红外。

红外唤醒包含两层：

- 云台负载上电。红外打开逻辑会依赖可见光云台 ready。
- 红外负载上电。带汇能精电等开关板时，会单独打开红外负载。

如果红外设备是高德 USB 设备，还会注册 USB 相关接收器，并请求 USB 权限。这样后续 `GUIDEDev` 才能识别和打开 USB 红外机芯。

进入 `GUIDEDev.open` 后，系统会设置打开中状态，并开始循环尝试打开红外机芯。

循环内每秒执行一次核心检查：

- 尝试打开红外机芯。
- 检查可见光云台是否 ready。
- 只有红外机芯打开成功，并且可见光云台 ready，才认为红外打开完成。

直播预览传入的自检等待参数为 false，因此红外直播预览不会额外补足 100 秒启动等待。只要红外和云台都 ready，就会立即进入直播启动。

如果在 120 秒超时时间内成功，系统清除打开中状态，回调打开成功。

打开成功后的直播回调与可见光类似：

- 更新 OSD 状态文本。
- 同步设备时间。
- 调用 `GUIDEDev.liveStart` 开启红外直播。
- 直播成功后刷新在线状态。

`GUIDEDev.liveStart` 会初始化视频编码器，启动红外预览，然后创建 RTP/H264 处理对象。随后启动一个处理线程，不断等待红外预览帧，把红外帧转成位图，绘制水印、测温区域、调色板、热点追踪和保护信息，再编码发送。

如果有直播测温区域，红外直播过程中还会按约 1 分钟周期上报一次区域温度数据。

如果 120 秒内红外没有成功打开，系统会认为红外打开失败。对于汇能精电控制器场景，`GUIDEDev` 会触发相机阻塞处理，倾向于重启红外负载；其他控制器场景则会回调打开失败。

打开失败后，直播入口会显示失败，结束直播任务，执行红外直播失败后的休眠处理，并尝试对红外负载进行下电再上电。

## 七、HuanYuDevice 与 GUIDEDev 的关键差异

`HuanYuDevice` 是可见光 ONVIF 设备，重点等待 ONVIF ready。它只需要云台负载上电。

`GUIDEDev` 是 USB 红外设备，重点等待 USB 红外机芯打开成功，同时还要求可见光云台 ready。它需要云台负载、红外负载和 USB 权限/USB 供电配合。

`HuanYuDevice` 直播启动后走 RTSP 拉流。

`GUIDEDev` 直播启动后走红外预览帧采集、图像处理、编码和 RTP 推送。

`HuanYuDevice` 直播预览打开成功后通常不会额外等待云台自检。

`GUIDEDev` 直播预览也不会额外补足 100 秒启动等待，但它本身会在打开循环中同时等待红外机芯和可见光云台 ready。

## 八、失败与保活处理

直播预览失败后，系统会结束直播任务，并根据设备类型执行对应休眠或重启负载动作。

可见光失败时，按可见光直播预览失败处理，可能会关闭可见光设备，并尝试重启云台负载。

红外失败时，按红外直播预览失败处理，可能会关闭红外设备，并尝试重启红外负载。

在唤醒模式下，直播预览成功或业务刚开始时都会刷新保活时间。业务结束后不会立即下电，而是等保活窗口结束后再次判断。

如果保活到期时仍在直播、打开、拍照、录像或回放，则继续保活。

如果保活到期时没有业务，并且附近没有定时拍照预热窗口，系统会关闭云台和红外，回到唤醒待机状态。

## 九、整体时序总结

可见光 `HuanYuDevice` 直播预览时序：

1. 主站请求可见光直播预览。
2. 唤醒模式下标记可见光直播业务。
3. 开启云台负载。
4. 创建或复用推流客户端。
5. 打开 ONVIF 设备，等待设备 ready。
6. 更新 OSD 和时间。
7. 启动 RTSP 直播。
8. 成功后刷新在线状态和唤醒保活。

红外 `GUIDEDev` 直播预览时序：

1. 主站请求红外直播预览。
2. 唤醒模式下标记红外直播业务。
3. 开启云台负载、红外负载和必要的 USB 支持。
4. 创建或复用推流客户端。
5. 打开高德红外机芯。
6. 同时等待红外机芯打开成功和可见光云台 ready。
7. 更新 OSD 和时间。
8. 启动红外预览、绘制水印和测温信息、编码推流。
9. 成功后刷新在线状态和唤醒保活。
