## 六、重点功能结果

### 1. 唤醒模式拍照后下电策略

| 场景 | 当前处理 |
| --- | --- |
| 可见光拍照完成 | 检查后续 10 分钟内是否有通道一或通道二定时拍照任务 |
| 红外拍照完成 | 检查后续 15 分钟内是否有通道一或通道二定时拍照任务 |
| 后续存在定时拍照 | 保持相关负载上电，等待后续任务 |
| 后续不存在定时拍照 | 拍照完成后执行下电 |
| 拍照启动阶段 | 日志区分为“拍照启动阶段临时保活”，避免和拍照完成后的下电策略混淆 |

### 2. 唤醒模式 MIPI 处理

| 场景 | 当前处理 |
| --- | --- |
| MIPI 有任务 | 任务开始前进行 MIPI 上电 |
| MIPI 任务结束 | 任务完成后进行 MIPI 下电 |
| 唤醒模式无 MIPI 任务 | MIPI 保持断电 |
| 休眠模式 | MIPI 一直断电 |

### 3. 直播启动中停止处理

| 场景 | 当前处理 |
| --- | --- |
| 启动过程中收到停止 | 标记延迟停止，不强制打断正在启动的设备打开流程 |
| 启动过程中又收到新拉流 | 复用当前启动过程服务新会话 |
| 启动完成后无新拉流 | 到安全点后执行停止和资源释放 |

### 4. 山火告警 2DH

| 场景 | 当前处理 |
| --- | --- |
| 山火告警短时间重复触发 | 合并重复上报，减少瞬间大量 2DH |
| 后台下发 2DH 查询或确认 | 补充确认回包处理，避免后台持续重复下发 |

### 5. OSD 和分辨率

| 场景 | 当前处理 |
| --- | --- |
| 微气象为空 | 按空数据分支处理 |
| 坐标为空 | 按空数据分支处理 |
| 微气象和坐标都为空 | 使用单独分支处理 |
| 320x240 等低分辨率拍照 | 使用自定义 OSD，避免机芯自带 OSD 看不清 |
| 其他分辨率 | 保持机芯自带 OSD 逻辑 |
| 高德红外环境温度 | 按控制器类型控制显示条件 |

## 七、当前阶段结论

从 `4932550` 到 `fc81774`，项目完成了一轮较集中的唤醒模式业务改造。核心成果是：

1. 唤醒模式下的设备上下电、业务保活和拍照后下电策略已经形成清晰规则。
2. 可见光、红外、MIPI 三类设备在拍照、拉流、短视频、录像回放等场景下的处理边界更加明确。
3. 红外启动慢、直播启动中停止、山火告警重复 2DH、OSD 空数据、低分辨率拍照 OSD 等问题均已完成阶段性处理。
4. 相关设计文档、分析文档和测试说明已经整理到 `说明文档` 目录，便于后续验证和交接。
