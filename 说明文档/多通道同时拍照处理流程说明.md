# 云台资源冲突处理流程说明

### 资源说明

    - 通道一：可见光机芯
    - 通道二：高德红外 

## 1. 问题描述

当前设备上，通道一和通道二在一些场景下会共用同一个物理云台。拍照、拉流中的预置位切换、人工云台控制等动作，最终都会影响云台实际位置。


## 2. 场景说明

当同一时刻出现多个通道拍照任务时，例如：

- 通道一调用预置位 1 拍照。
- 通道二调用预置位 2 拍照。

软件不会让两个通道同时转动云台并同时抓拍，而是把多个拍照任务按到达顺序排队处理。

这里的拍照任务包括后台下发的手动拍照，也包括设备本地触发的定时拍照。无论拍照来源是哪一种，只要任务触发，处理流程都会按当前队列规则执行。

## 3. 当前处理原则

当前程序的实际处理原则是：

1. 先收到的拍照任务先处理。
2. 后收到的拍照任务等待前一个拍照任务完成。
3. 每个拍照任务都会按自己的通道和预置位独立执行。
4. 一个拍照任务完成后，再执行下一个拍照任务。
5. 多张照片会按各自的通道号、预置位和时间生成文件并进入上传流程。

因此，同一时刻下发的多通道拍照，在设备上实际表现为顺序执行，而不是并行执行。


## 4. 拍照与拉流预置位冲突

当前队列主要解决的是多个拍照任务之间的先后顺序。

云台本身的动作是有先后顺序的。例如先调用预置位 1，再马上调用预置位 2，云台会先转到预置位 1，再继续转到预置位 2。

如果在拉流过程中发起拍照，并且拍照需要调用另一个预置位，就可能出现以下场景：

1. 当前正在拉流，画面停留在预置位 1。
2. 后台或定时任务发起拍照，要求调用预置位 2。
3. 云台开始转向预置位 2，准备拍照。
4. 照片还没有真正抓拍完成时，又有人在视频中调用预置位 1。
5. 云台又开始转回预置位 1。
6. 此时拍照继续执行，抓到的画面不是预置位 2 的稳定画面。

这种情况下，照片可能出现以下问题：

1. 拍到云台转动过程中的画面。
2. 拍到两个预置位之间的中间位置。
3. 照片对应的预置位信息和实际画面不一致。

## 5. 拍照与拉流预置位冲突解决方案

### 5.1 处理目标

解决这个问题的目标是：拍照调用预置位后，到照片抓拍完成前，不允许在视频中或人工操作把云台转到其他预置位。

也就是说，拍照期间云台位置要被临时保护，保证照片实际画面和拍照任务要求的预置位一致。

### 5.2 处理原则

采用“拍照优先，视频预置位延后”的策略。

处理原则如下：

1. 拍照任务一旦开始调用预置位，就进入拍照保护状态。
2. 拍照保护状态下，视频中的预置位切换不立即执行。
3. 如果拍照期间用户多次点击视频预置位，只保留最后一次点击的预置位。
4. 拍照完成后，先结束拍照保护状态。
5. 如果拍照期间记录了视频中最后一次预置位请求，再把云台转到这个预置位。

这样可以保证：

1. 拍照不会被视频侧预置位打断。
2. 用户在视频侧最后想看的预置位不会丢失。
3. 不需要对所有云台动作做复杂优先级仲裁。
4. 多次点击不会排出一长串云台动作，只执行最后一次有效操作。

### 5.3 流程

以“拉流在预置位 1，拍照要求预置位 2”为例，推荐流程如下：

1. 当前视频正在拉流，画面停在预置位 1。
2. 拍照任务开始，进入拍照保护状态。
3. 云台转到预置位 2。
4. 等待云台稳定。
5. 抓拍照片。
6. 抓拍完成，退出拍照保护状态。
7. 如果拍照期间用户点击过视频预置位 1，则拍照完成后再转回预置位 1。
8. 视频画面恢复到用户最后选择的预置位。


### 5.4 异常场景处理

如果拍照失败，也应该退出拍照保护状态。

拍照失败后，如果拍照期间有视频侧预置位请求，可以继续执行最后一次视频预置位请求，避免云台长时间停在中间位置或拍照目标位置。


### 5.5 修改建议

增加一个全局拍照保护状态和一个待执行的视频预置位记录。

修改逻辑如下：

1. 拍照开始调用预置位前，标记“拍照保护中”。
2. 拍照成功、拍照失败或拍照超时后，清除“拍照保护中”。
3. 视频侧调用预置位时，如果正在拍照保护中，不立即转动云台，只记录最后一次请求。
4. 拍照保护结束后，如果存在待执行的视频预置位，再执行这一次转动。
