# SPGProtocol 指令处理总结

本文根据 `app/src/main/java/hikvision/zhanyun/com/hikvision/SPGProtocol.java` 中的控制字分发逻辑整理，说明项目收到各类指令后执行了哪些操作。

当前先整理 `00H` 到 `48H`。后续可继续补充图片、视频、AI、升级、日志、红外测温等指令。

## 一、基础通信类指令

| 指令 | 名称 | 项目中的处理 |
| --- | --- | --- |
| `00H` | 开机联络 | 如果收到空数据域，设备主动上送版本号 `VERSION=V3.0`。如果收到主站响应，则标记 `Logged=true`，主动发起 `01H` 校时，并回调 `listenerCallBack.onLoggedIn()`。 |
| `01H` | 校时 | 解析主站下发的年月日时分秒，原命令确认返回，然后调用 `listenerCallBack.setClock(...)` 设置本地时间。设备开机后也会主动发送空数据域 `01H` 请求校时。 |
| `02H` | 设置密码 | 校验旧密码，正确则取后 4 字节作为新密码，调用 `listenerCallBack.setPassword(...)` 保存，并原样确认；错误返回 `FFFF`。 |
| `03H` | 下发参数 | 校验密码后解析心跳间隔、采样间隔、休眠时长、在线时长、重启时间、密文认证，调用 `listenerCallBack.setOnlineConfig(...)`。 |
| `04H` | 未处理 | `switch` 中为空分支，当前项目没有实际动作。 |
| `05H` | 心跳 | 收到主站心跳响应时调用 `listenerCallBack.onHeartBeatReceived()`；设备主动心跳由 `doHeartBeat(...)` 组包发送，包含时间、信号、电压。 |

## 二、服务器与设备控制类指令

| 指令 | 名称 | 项目中的处理 |
| --- | --- | --- |
| `06H` | 设置服务器地址 | 校验密码，比较两组 IP、端口、卡号是否一致；一致则调用 `listenerCallBack.changeServer(server, port)`，并更新 SIM；不一致返回 `{0,0}`。 |
| `07H` | 查询服务器设置 | 返回当前主站 IP、端口、SIM 卡号。SIM 不足 12 位左侧补 `F`，超过则截断。 |
| `08H` | 重启 | 校验密码，正确则原样确认并调用 `listenerCallBack.rebootDevice()`。 |
| `09H` | 唤醒 | 当前代码没有校验密码，直接 `echoBack(09H)`，然后调用 `listenerCallBack.wakeup()`。 |
| `0AH` | 查询基本配置 | 调用 `listenerCallBack.getConfigurationParameters()`，将 `Settings.Parameters` 打包返回。 |
| `0BH` | 功能配置 | 校验密码后创建 `Settings.Features` 并调用 `listenerCallBack.setFeatures(features)`；当前不回包。 |
| `0CH` | 休眠通知/休眠相关 | 接收时当前不是直接 `doSleep()`，而是调用 `listenerCallBack.ptzControl(1, 18, 5)`，然后回显 `0CH`。另外 `sleep(int channel)` 可主动发送 `0CH` 休眠通知。 |

| `0EH` | 确认短信 | 校验密码，解析 SIM 卡号，调用 `listenerCallBack.sendSMS(sim, deviceID)`，并原样确认。 |

### 通用文件上传流程说明

`71H~76H` 是一套通用文件上传流程，与后面的图片/短视频上传不同。

+ `71H`：主站先查设备有哪些文件可上传。
+ `72H`：主站指定设备开始上传。
+ `73H`：设备上报即将上传的文件名、采样时间、文件长度、总包数。
+ `74H`：设备按 `MAX_UPLOAD_FILE_SIZE` 分包上传文件内容。
+ `75H`：设备发送文件上传结束帧。
+ `76H`：主站发现缺包时，指定包号要求设备补传。

### 图片上传流程说明

图片上传不是主站直接拉文件，而是设备先调用 `doUploadFile(time, fileName, channel, preset, FILE_TYPE.PHOTO)` 主动发起：

+ 设备检查文件是否存在、是否还在写入、是否为空。
+ 设置上传锁 `uploading=true`，避免多个文件同时上传。
+ 通过 `84H` 通知主站有图片要上传，最多重试 5 次。
+ 主站响应 `84H` 后，设备用 `85H` 按包发送图片内容。
+ 设备发送完后，用 `86H` 发送结束帧，最多重复发送直到主站回应。
+ 主站若发现缺包，下发 `87H`，设备按包号补传。
+ 如果 `87H` 中缺包数为 0，设备认为上传成功，调用 `listenerCallBack.onFileUploadEnd(...)`。

### 短视频上传流程说明

短视频上传和图片上传共用同一个 `doUploadFile(...)` 分包上传框架，只是控制字不同：

+ 短视频上传请求：`94H`
+ 短视频内容分包：`95H`
+ 短视频上传结束：`96H`
+ 短视频补包：`97H`

`93H` 只负责“开始录制短视频”，录制完成后才会由业务层调用 `doUploadFile(..., FILE_TYPE.VIDEO)` 发起 `94H` 上传。

### 录像回放和录像下载流程说明

当前 `9AH` 和 `9DH` 使用同一个解析方法 `doPlaybackStream(...)`：

+ 自动判断新协议或旧协议。
+ 解析服务器 IP、端口、录像起止时间、SSRC。
+ 调用 `listenerCallBack.playbackFile(...)`。
+ 回放使用 `ORDER_9AH` 应答。
+ 下载使用 `ORDER_9DH` 应答。

这样设计的原因是：服务器下发的录像下载 `9DH` 与录像回放 `9AH` 参数结构一致，设备端本质上都是“按时间段取录像并推送到服务器”。区别只是服务器侧把这路流用于播放还是保存。
