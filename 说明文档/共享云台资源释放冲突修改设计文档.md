# 共享云台资源释放冲突修改设计文档

## 1. 背景

当前设备中，可见光机芯和红外机芯公用同一个云台。可见光拉流、拍照、录制短视频、录像回放都依赖云台可见光机芯负载；红外拉流、拍照、录制短视频也需要先保证云台可见光机芯负载可用，再开启红外负载。

现有释放逻辑主要按当前业务或当前通道结束来判断是否下电，容易出现“某一路业务结束，把另一路仍在使用的共享负载关掉”的问题。尤其在可见光回放和红外业务并发时，红外业务结束后的释放动作可能影响可见光回放、可见光拉流或可见光拍照。

本文只描述逻辑关系和设计原则，不添加具体代码实现。

## 2. 当前逻辑关系分析

### 2.1 关键负载关系

| 负载 | 业务含义 | 共享关系 | 释放风险 |
| --- | --- | --- | --- |
| 云台可见光机芯负载 | 云台、可见光机芯、可见光通道能力 | 可见光和红外共同依赖 | 任一业务结束时误关，会影响另一业务 |
| 红外负载 | 红外机芯及其必要支撑能力 | 只供红外业务使用 | 红外结束后可独立释放，但不能带着云台一起误释放 |
| MIPI 负载 | MIPI 摄像头独立链路 | 不参与云台共享 | 只影响 MIPI 自身业务 |
| 网络共享链路 | 拉流、拍照、短视频、回放的网络支撑 | 可被多类业务复用 | 不能仅按单个业务结束直接关闭 |

### 2.2 当前释放冲突

| 场景 | 当前逻辑风险 | 正确逻辑关系 |
| --- | --- | --- |
| 可见光回放中，红外拉流结束 | 红外释放路径可能关闭云台可见光机芯负载 | 只释放红外独占负载，保留云台可见光机芯负载 |
| 可见光拉流中，红外拍照结束 | 红外拍照完成后可能触发云台释放 | 可见光拉流仍占用云台，云台不能释放 |
| 红外短视频中，可见光拍照完成 | 可见光拍照完成后可能误判云台空闲 | 红外短视频仍依赖云台，云台不能释放 |
| 可见光回放停止后，无其他业务 | 回放释放后如果无保活、无预热、无其他占用，云台才可释放 | 释放必须看全局占用关系 |
| 基础模式保活期内业务结束 | 单个业务结束后可能立即下电 | 保活期未结束前不允许释放共享负载 |
| MIPI 业务结束 | 可能被错误纳入云台释放判断 | MIPI 独立释放，不影响云台和红外 |

### 2.3 主要问题归纳

| 问题 | 说明 |
| --- | --- |
| 释放中心不正确 | 当前更偏向“通道结束即释放”，缺少“共享负载是否仍被占用”的全局判断 |
| 可见光回放保护不足 | 回放属于可见光业务，也必须被视为云台可见光机芯负载占用者 |
| 红外释放粒度过粗 | 红外结束时应优先释放红外独占负载，再单独判断云台是否可释放 |
| 保活只能延迟冲突 | 基础模式保活可以减少频繁下电，但不能替代共享资源占用判断 |
| MIPI 边界需要明确 | MIPI 是独立链路，不应参与云台可见光机芯和红外的共享占用关系 |

## 3. 修改目标

| 目标 | 说明 |
| --- | --- |
| 共享云台不被误释放 | 任何业务释放云台前，必须确认可见光和红外都没有业务继续占用它 |
| 红外可以单独释放 | 红外业务结束后，如果可见光仍在使用云台，只释放红外独占负载 |
| 可见光释放不影响红外 | 可见光业务结束后，如果红外仍在执行，云台可见光机芯负载必须保留 |
| 回放纳入占用保护 | 可见光录像回放期间，其他业务结束不能关闭云台可见光机芯负载 |
| MIPI 独立释放 | MIPI 拉流、拍照、短视频结束只影响 MIPI 负载，不影响云台和红外 |
| 保留基础模式策略 | 基础模式 15 分钟保活、拍照和短视频后的后续定时拍照窗口继续有效 |

## 4. 资源模型设计

### 4.1 资源分类

| 资源 | 含义 | 归属关系 |
| --- | --- | --- |
| 云台可见光机芯资源 | 云台、可见光机芯、可见光通道能力 | 可见光和红外共享 |
| 红外资源 | 红外机芯及红外业务能力 | 红外独占 |
| 红外支撑资源 | 红外所需的 USB 或其他辅助能力 | 红外独占 |
| 网络共享资源 | 拉流、拍照、短视频、回放的网络通路 | 多业务复用 |
| MIPI 资源 | MIPI 摄像头链路 | MIPI 独立 |

### 4.2 业务资源需求

| 业务 | 需要占用的资源 | 释放原则 |
| --- | --- | --- |
| 可见光拉流 | 云台可见光机芯资源、网络共享资源 | 拉流停止后只释放自身占用，云台是否关闭看全局 |
| 可见光拍照 | 云台可见光机芯资源、网络共享资源 | 拍照完成后先判断后续定时拍照窗口，再进入共享释放判断 |
| 可见光短视频 | 云台可见光机芯资源、网络共享资源 | 短视频队列结束后释放自身占用 |
| 可见光录像回放 | 云台可见光机芯资源、网络共享资源 | 回放期间必须保护云台不被其他业务释放 |
| 红外拉流 | 云台可见光机芯资源、红外资源、红外支撑资源、网络共享资源 | 红外停止后先释放红外独占资源，再判断云台是否仍被占用 |
| 红外拍照 | 云台可见光机芯资源、红外资源、红外支撑资源、网络共享资源 | 拍照完成后不能直接同时关闭红外和云台 |
| 红外短视频 | 云台可见光机芯资源、红外资源、红外支撑资源、网络共享资源 | 短视频结束后红外可独立释放，云台需全局判断 |
| MIPI 拉流、拍照、短视频 | MIPI 资源 | 只释放 MIPI 资源，不影响云台和红外 |

### 4.3 资源占用记录逻辑

业务开始时记录“谁占用了哪些资源”，业务结束时只释放自己的占用。是否真正下电，不由单个业务直接决定，而由当前全部占用关系共同决定。

| 记录项 | 含义 |
| --- | --- |
| 占用对象 | 本次业务占用的资源集合 |
| 业务类型 | 拉流、拍照、短视频、回放、预热、保活 |
| 业务通道 | 可见光、红外、MIPI 或对应通道 |
| 开始时间 | 用于判断占用时长和异常兜底 |
| 释放原因 | 用于现场排查是哪个业务触发释放 |

## 5. 释放规则

### 5.1 云台可见光机芯负载释放规则

只有同时满足以下条件，才允许关闭云台可见光机芯负载：

| 条件 | 要求 |
| --- | --- |
| 资源占用 | 没有任何可见光或红外业务占用云台可见光机芯资源 |
| 业务状态 | 没有拉流、拍照、短视频、回放、打开中、重试中的相关业务 |
| 基础模式保活 | 不处于基础模式 15 分钟保活期 |
| 可见光预热窗口 | 不处于可见光拍照前 3 分钟预热窗口 |
| 红外预热窗口 | 不处于红外拍照前 10 分钟预热窗口 |
| 回放保护 | 可见光录像回放已经完全停止并释放占用 |

### 5.2 红外负载释放规则

只有同时满足以下条件，才允许关闭红外负载：

| 条件 | 要求 |
| --- | --- |
| 红外占用 | 没有任何红外拉流、拍照、短视频占用红外资源 |
| 红外状态 | 红外没有打开中、执行中、重试中、收尾中的业务 |
| 红外预热窗口 | 不处于红外拍照前 10 分钟预热窗口 |
| 共享云台判断 | 红外释放不代表云台可释放，云台必须单独判断 |

红外业务结束时的正确结果应为：红外独占资源可以释放；云台可见光机芯负载只有在可见光和红外都不再占用、且不处于保活或预热窗口时，才允许释放。

### 5.3 MIPI 释放规则

| 条件 | 要求 |
| --- | --- |
| MIPI 占用 | 没有 MIPI 拉流、拍照、短视频正在执行 |
| MIPI 状态 | MIPI 不处于打开中、执行中、重试中 |
| 影响范围 | MIPI 释放只影响 MIPI 自身，不触发云台和红外释放 |

## 6. 业务流程调整

### 6.1 业务开始流程

| 步骤 | 处理 |
| --- | --- |
| 1 | 根据业务类型和通道确认需要占用的资源 |
| 2 | 记录本次业务对相关资源的占用关系 |
| 3 | 按资源依赖顺序保证负载可用 |
| 4 | 启动拉流、拍照、短视频或回放业务 |
| 5 | 如果启动失败，释放本次业务占用，再进入统一释放判断 |

### 6.2 业务结束流程

| 步骤 | 处理 |
| --- | --- |
| 1 | 先结束业务自身状态，例如停止拉流、结束拍照、结束短视频、停止回放 |
| 2 | 释放本次业务登记的资源占用 |
| 3 | 如果处于基础模式，先判断保活期和后续定时拍照窗口 |
| 4 | 分别判断红外负载、云台可见光机芯负载、MIPI 负载是否满足释放条件 |
| 5 | 只释放满足条件的负载，不满足条件的负载继续保持 |

### 6.3 释放关系原则

| 原则 | 说明 |
| --- | --- |
| 先释放独占，再判断共享 | 红外业务结束后，先处理红外独占资源，再判断云台共享资源 |
| 谁占用谁释放 | 单个业务只能释放自己的占用记录，不能代表其他业务释放共享负载 |
| 共享资源全局判断 | 云台可见光机芯负载必须确认所有占用都结束后才能关闭 |
| 保活和预热优先保护 | 基础模式保活期、拍照预热窗口内不允许释放相关负载 |
| 回放作为正式占用 | 可见光回放与拉流、拍照、短视频一样，都属于云台占用者 |
| MIPI 独立边界 | MIPI 的开始和结束不改变云台可见光机芯和红外的释放判断 |

## 7. 需要纳入统一判断的业务出口

| 业务出口 | 统一判断要求 |
| --- | --- |
| 拉流开始失败 | 释放本次拉流占用，再判断是否允许下电 |
| 拉流停止 | 释放本次拉流占用，再判断共享负载 |
| 拍照成功 | 先处理后续定时拍照窗口，再判断共享负载 |
| 拍照失败 | 释放本次拍照占用，再判断共享负载 |
| 补拍结束 | 释放补拍相关占用，再判断共享负载 |
| 短视频单条完成 | 如果后面还有队列任务，继续保持对应占用 |
| 短视频队列结束 | 释放短视频占用，再判断共享负载 |
| 短视频失败 | 释放失败任务占用，再判断共享负载 |
| 可见光回放开始 | 记录可见光回放对云台可见光机芯资源的占用 |
| 可见光回放停止 | 释放回放占用，再判断云台是否可关闭 |
| 基础模式保活到期 | 保活到期后仍需再次检查所有资源占用关系 |
| MIPI 业务结束 | 只判断 MIPI 负载，不影响云台和红外 |

## 8. 状态和互斥补充

资源占用关系解决的是“能不能下电”的问题，不等于解决所有业务互斥问题。

| 类型 | 规则 |
| --- | --- |
| 电源资源共享 | 可见光和红外可以同时占用云台可见光机芯资源，只要任一占用存在就不能关闭 |
| 设备操作互斥 | 同一设备不应同时执行互斥操作，例如同一通道同时回放和拉流 |
| 云台动作互斥 | 如果业务会转预置位、巡航或移动云台，需要避免一个业务移动云台影响另一个业务画面 |
| MIPI 独立 | MIPI 不参与云台可见光机芯和红外的资源占用计数 |

本次建议优先解决电源资源共享保护。云台动作互斥可作为第二阶段增强。

## 9. 典型场景

### 9.1 可见光回放中，红外拉流结束

| 时间 | 事件 | 期望处理 |
| --- | --- | --- |
| T0 | 可见光回放开始 | 占用云台可见光机芯资源 |
| T1 | 红外拉流开始 | 占用云台可见光机芯资源、红外资源、红外支撑资源 |
| T2 | 红外拉流结束 | 释放红外拉流自身占用 |
| T3 | 资源判断 | 红外资源可关闭；云台仍被可见光回放占用，不能关闭 |
| T4 | 可见光回放继续 | 回放不被红外释放动作打断 |

### 9.2 红外短视频中，可见光拍照完成

| 时间 | 事件 | 期望处理 |
| --- | --- | --- |
| T0 | 红外短视频开始 | 占用云台可见光机芯资源、红外资源、红外支撑资源 |
| T1 | 可见光拍照开始 | 共享占用云台可见光机芯资源 |
| T2 | 可见光拍照完成 | 释放可见光拍照占用 |
| T3 | 资源判断 | 云台仍被红外短视频占用，不能关闭 |
| T4 | 红外短视频继续 | 不受可见光拍照释放影响 |

### 9.3 红外拍照完成，后续无任务

| 时间 | 事件 | 期望处理 |
| --- | --- | --- |
| T0 | 红外拍照开始 | 占用云台可见光机芯资源、红外资源、红外支撑资源 |
| T1 | 红外拍照完成 | 释放红外拍照占用 |
| T2 | 后续窗口检查 | 无后续定时拍照、无保活、无预热窗口 |
| T3 | 资源判断 | 红外资源关闭；云台无其他占用时关闭 |

### 9.4 MIPI 拉流结束

| 时间 | 事件 | 期望处理 |
| --- | --- | --- |
| T0 | MIPI 拉流开始 | 占用 MIPI 资源 |
| T1 | 可见光或红外业务运行中 | 云台和红外按各自占用关系保持 |
| T2 | MIPI 拉流结束 | 释放 MIPI 资源 |
| T3 | 资源判断 | 不触发云台可见光机芯负载和红外负载释放 |

## 10. 验收用例

| 用例 | 操作 | 期望结果 |
| --- | --- | --- |
| 可见光回放中停止红外拉流 | 先开始可见光回放，再开始红外拉流，然后停止红外拉流 | 红外可以停止和下电，可见光回放不中断，云台可见光机芯不下电 |
| 可见光拉流中红外拍照完成 | 可见光持续拉流，触发红外拍照并完成 | 红外按规则释放，可见光拉流不中断 |
| 红外短视频中可见光拍照完成 | 红外录制短视频时触发可见光拍照 | 可见光拍照完成后不关闭云台，红外短视频继续 |
| 红外结束后无可见光业务 | 红外拉流或拍照结束，当前没有其他业务和预热窗口 | 红外负载关闭；云台按基础模式保活和预热窗口规则关闭 |
| 可见光回放停止后无其他业务 | 停止可见光回放，当前无红外业务、无保活、无预热窗口 | 云台可见光机芯允许关闭 |
| 基础模式保活期内释放 | 任一业务结束后仍处于保活期 | 不下电，保活到期后再按资源占用判断 |
| MIPI 业务结束 | MIPI 拉流、拍照或短视频结束 | 只影响 MIPI 负载，不影响云台和红外 |

## 11. 实施顺序建议

| 阶段 | 内容 |
| --- | --- |
| 第一阶段 | 统一梳理拉流、拍照、短视频、回放、保活、预热对资源的占用关系 |
| 第二阶段 | 先只记录占用关系和释放原因，用现场日志验证模型是否符合实际业务 |
| 第三阶段 | 将拉流、拍照、短视频、回放的结束出口纳入统一释放判断 |
| 第四阶段 | 所有下电动作都先经过共享资源判断，禁止单个业务直接决定共享负载下电 |
| 第五阶段 | 补充云台动作互斥，处理预置位、巡航、巡检等可能互相影响画面的操作 |

| 检查点 | 需要记录的信息 |
| --- | --- |
| 业务开始 | 业务类型、通道、占用资源、开始原因 |
| 业务结束 | 业务类型、通道、释放资源、结束原因 |
| 拒绝下电 | 当前仍占用该负载的业务、保活状态、预热窗口状态 |
| 允许下电 | 负载类型、确认无占用的结果、触发原因 |
| 异常兜底 | 占用持续时间、业务最后状态、兜底释放原因 |

## 13. 结论

可见光机芯和红外机芯公用云台可见光机芯负载，不能再由单个业务结束路径直接决定云台是否下电。需要在业务开始和结束之间建立共享资源占用关系，让所有下电动作先经过全局资源判断。

修改后，红外业务释放只能关闭自己独占的红外负载；云台可见光机芯负载必须等可见光拉流、可见光拍照、可见光短视频、可见光回放以及红外业务都释放后，并且不处于基础模式保活或预热窗口时，才允许关闭。MIPI 业务保持独立释放，不影响云台和红外。
