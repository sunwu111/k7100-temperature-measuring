# 共享云台下拍照与拉流预置位仲裁设计方案

## 1. 背景

当前项目中，可见光机芯和红外机芯共用同一个物理云台。可见光拍照、拉流、短视频、录像回放都依赖云台；红外虽然图像来自红外机芯，但拍照、拉流和测温区域也依赖同一个云台方位。

所以这里有两个问题：

1. 共享负载问题：业务结束时，不能误关另一个业务仍在使用的云台或红外负载。
2. 共享动作问题：拍照和拉流都会调用预置位时，不能让一个业务随意转动云台，影响另一个业务画面。

当前项目已经有共享负载保护逻辑，但共享动作仲裁还不够完整。

## 2. 当前项目是如何处理的

### 2.1 拍照预置位

拍照指令中已经带有通道号和预置位号。当前流程大致是：

1. 主站或定时任务触发拍照。
2. 系统根据通道找到可见光或红外设备。
3. 唤醒模式下刷新保活。
4. 可见光拍照前保证云台负载上电。
5. 红外拍照前保证云台和红外负载都上电。
6. 设备打开后，由具体机芯自己转到拍照预置位，等待云台稳定后抓图。

也就是说，拍照预置位目前是“各机芯自己处理”。可见光和红外都能在拍照前调用预置位，但这个动作没有经过统一仲裁。

### 2.2 拉流预置位

当前远程直播启动主要处理上电、打开设备、建立推流连接和开始推流。

远程直播指令当前没有独立的直播预置位字段，因此设备无法单靠直播指令知道“拉流前要转到哪个预置位”。

本地预览有预置位参数，可以在预览时调用预置位，但这和远程直播不是同一套完整机制。

如果后台要求“拉流调用预置位”，当前只能采用两种方式：

1. 后台先下发云台预置位指令，再下发直播指令。
2. 后续扩展直播指令，让直播指令本身携带预置位。

### 2.3 共享负载释放

项目里已经有一些保护：

1. 云台负载被视为可见光和红外共同依赖的资源。
2. 红外业务额外依赖红外负载。
3. 唤醒模式下，直播、回放、拍照、短视频会刷新保活。
4. 下电前会检查当前是否还有业务忙。
5. 下电前会检查是否接近定时拍照预热窗口。
6. 拍照或短视频完成后，会根据后续定时拍照任务决定继续保活还是下电。

这部分解决的是“不要误下电”的问题。

## 3. 当前存在的问题

### 3.1 预置位动作分散

拍照、预览、巡航、巡检、录像预置位等业务都可能直接移动云台。每个业务只知道自己的目标预置位，不知道其他业务是否正在使用云台。

### 3.2 直播画面可能被拍照转走

如果后台正在看直播，此时又触发另一个预置位的拍照，拍照流程可能直接把云台转到拍照预置位。这样直播画面会被改变。

### 3.3 红外和可见光可能抢同一个云台

红外拍照、红外拉流、可见光拍照、可见光拉流看起来是不同通道，但实际使用同一个云台。只要目标预置位不同，就不能真正并行移动。

### 3.4 拍照后恢复位置不统一

拍照完成后，云台应该停在拍照预置位、恢复直播预置位，还是恢复录像预置位，目前没有统一规则。

### 3.5 远程直播预置位来源不清

当前远程直播指令没有明确预置位字段。后台如果希望“转到某个预置位再拉流”，需要补充协议或操作约定。

## 4. 新方案

### 4.1 增加共享云台动作仲裁

建议在业务层增加统一的云台动作仲裁逻辑。

所有会移动云台的业务，都先申请云台动作，不能各自直接移动。仲裁逻辑只负责判断三件事：

1. 当前云台能不能动。
2. 多个业务都要动云台时，谁先动。
3. 临时动作完成后，云台要不要恢复到之前的位置。

电源管理仍然按现有逻辑处理。也就是说，先保证负载上电，再申请云台动作；业务结束后，再按现有保活和下电规则判断是否释放负载。

### 4.2 基本规则

| 场景 | 建议处理 |
| --- | --- |
| 没有其他业务时拍照 | 转到拍照预置位，到位后拍照 |
| 没有其他业务时拉流 | 如果有直播预置位，先转预置位再拉流；没有预置位则按当前位置拉流 |
| 直播中同预置位拍照 | 复用当前云台位置，不重复转动 |
| 直播中不同预置位拍照 | 普通定时拍照不打断直播；告警拍照或主站允许时可临时打断 |
| 拍照中收到拉流 | 等拍照完成后再启动拉流 |
| 红外和可见光同时请求不同预置位 | 按优先级串行执行，不能并行移动 |
| 手动云台控制 | 优先级高，执行后不自动恢复 |

### 4.3 优先级建议

从高到低建议如下：

1. 手动停止、急停、停止巡航、停止巡检。
2. 主站手动云台控制。
3. 告警联动拍照。
4. 主站手动请求拍照。
5. 直播启动前调用预置位。
6. 短视频录制前调用预置位。
7. 定时拍照、定时录像调用预置位。
8. 自动恢复录像预置位。
9. 巡航、巡检。

高优先级任务可以排在前面，但是否允许打断当前业务，需要根据业务类型决定。比如普通定时拍照不建议打断直播，告警拍照可以允许临时打断。

### 4.4 恢复规则

| 场景 | 建议恢复 |
| --- | --- |
| 直播中临时拍照 | 拍照完成后恢复直播预置位 |
| 工作时间有录像预置位 | 空闲后恢复录像预置位 |
| 巡航或巡检被打断 | 根据配置决定继续还是结束 |
| 手动云台控制 | 不自动恢复 |
| 唤醒模式无后续任务 | 不恢复，直接进入下电判断 |

### 4.5 直播预置位来源

推荐优先采用“扩展直播指令”的方式，让直播指令携带预置位。

如果暂时不改协议，可以先采用兼容方案：后台先下发云台预置位指令，设备确认或等待到位后，再下发直播指令。

无论采用哪种方式，设备侧仍需要云台动作仲裁。不能只依赖后台命令顺序，因为设备端还可能同时遇到定时拍照、告警拍照、巡航、巡检等任务。

## 5. 实施建议

建议分三步做：

1. 先梳理所有会移动云台的入口，补充日志，确认现场到底是谁在转云台。
2. 再增加统一互斥，保证同一时间只有一个业务能移动云台。
3. 最后增加优先级、直播保持和恢复策略。

短期可以先解决最容易出问题的场景：

1. 直播中普通定时拍照不要直接转走云台。
2. 直播中告警拍照允许临时转动，但完成后恢复直播预置位。
3. 拍照过程中收到拉流，先等拍照完成再拉流。
4. 红外和可见光请求不同预置位时必须串行。

## 6. 结论

当前项目已经对共享云台的上电、保活和下电做了一定保护，可以减少业务结束后误下电的问题。

但拍照和拉流都调用预置位时，核心矛盾变成“谁能移动云台”。当前预置位动作分散在多个业务里，没有统一仲裁，所以可能出现直播画面被拍照转走、红外和可见光抢云台、拍照后位置恢复不一致等问题。

建议新增共享云台动作仲裁：所有预置位动作统一排队、统一优先级、统一恢复策略；同时明确远程直播预置位来源。这样可以在不破坏现有电源管理策略的前提下，保证拍照结果正确、直播画面稳定。
