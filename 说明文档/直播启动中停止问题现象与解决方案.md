# 直播启动中停止问题现象与解决方案

## 一、问题现象

在唤醒模式下和正常模式下，可见光或红外执行直播拉流时，存在一种开始直播和停止直播交错的情况。

典型现象如下：

| 序号 | 现象 | 说明 |
| --- | --- | --- |
| 1 | 后台下发开始直播后，设备还在启动中 | 此时设备可能正在上电、等待云台 ready、等待红外 USB 识别、打开机芯或初始化编码器 |
| 2 | 设备还没有稳定出流，后台或用户又下发停止直播 | 停止指令可能来自后台等待超时，也可能来自用户手动停止 |
| 3 | 停止指令如果直接打断启动流程 | 可能导致设备刚打开又被关闭，出现开始和停止状态交错 |
| 4 | 停止指令如果不响应 | 后台会一直等待，用户侧也可能认为设备没有处理停止 |
| 5 | 用户只手动停止一次，后面不再补发停止指令 | 如果设备内部没有记录停止请求，设备启动成功后可能继续保持直播 |
| 6 | 启动中停止后又有新的 SSRC 拉流 | 如果没有复用逻辑，旧停止可能把新的拉流停止掉，或者设备重复打开 |

## 二、问题本质

这个问题不是单纯的红外启动慢，也不是简单加长超时时间就能解决。

核心问题是：直播启动流程和停止流程没有统一的状态管理。

| 当前缺口 | 影响 |
| --- | --- |
| 只判断是否已经直播，没有判断是否正在启动 | 启动中还没有进入直播状态，停止逻辑无法正确区分 |
| 停止直播可能直接关闭设备 | 容易打断正在打开的红外或可见光 |
| 停止请求没有内部保留 | 用户只停止一次时，启动成功后可能没人再停 |
| 新 SSRC 拉流没有接管启动流程 | 旧停止和新拉流容易互相影响 |

## 三、处理原则

正确处理方式不是“不响应停止”，而是把停止分成两层。

| 层级 | 处理原则 |
| --- | --- |
| 命令响应层 | 收到停止直播后，立即给后台返回停止响应 |
| 设备执行层 | 如果设备仍在启动中，不立即强制中断，而是记录延迟停止请求 |

也就是说：

停止命令必须立即响应。

设备内部可以延迟收口。

停止请求必须被记录，不能丢失。

## 四、红外处理方案

红外启动过程中，如果收到停止直播，建议按下面逻辑处理。

| 场景 | 处理 |
| --- | --- |
| 红外正在启动中，收到停止直播 | 立即响应停止指令 |
| 红外还在上电、USB 识别、打开机芯或准备出流 | 不立即强制关闭，记录延迟停止请求 |
| 后续没有新的红外拉流 | 启动成功、失败、超时或到达安全点后，自动执行停止 |
| 后续收到新的红外拉流 | 取消旧的延迟停止，复用当前启动流程，服务新的 SSRC |

红外这里尤其要避免启动过程中直接强制关闭，因为红外启动链路比可见光更长，包含云台、可见光 ready、红外上电、USB 识别、红外机芯打开、编码器初始化等阶段。

## 五、可见光处理方案

可见光和红外的处理原则一致。

| 场景 | 处理 |
| --- | --- |
| 可见光正在启动中，收到停止直播 | 立即响应停止指令 |
| 可见光还在等待云台 ready、打开球机或准备出流 | 不立即强制关闭，记录延迟停止请求 |
| 后续没有新的可见光拉流 | 到达安全点后自动停止 |
| 后续有新的 SSRC 拉流 | 旧停止作废，复用当前启动流程，服务新的 SSRC |

可见光需要重点处理新 SSRC 接管问题，避免旧的停止指令把新的可见光拉流停掉。

## 六、新拉流复用规则

当设备处于启动中时，如果又收到同通道新的拉流请求，不建议重新打开设备。

推荐规则如下：

| 条件 | 处理 |
| --- | --- |
| 同一通道正在启动 | 不重复打开设备 |
| 新拉流 SSRC 与旧 SSRC 不同 | 更新当前服务 SSRC |
| 之前已经收到停止请求 | 取消旧的延迟停止请求 |
| 当前启动成功 | 直接服务新的 SSRC |
| 当前启动失败或超时 | 返回新拉流失败，清理启动状态 |

这样可以避免两类问题：

| 问题 | 解决方式 |
| --- | --- |
| 旧停止误停新拉流 | 新拉流到来后取消旧延迟停止 |
| 重复打开设备 | 新拉流复用当前启动流程 |

## 七、没有新拉流时如何停止

如果启动过程中收到停止直播，后续没有新的拉流请求，设备不能继续保持直播。

推荐处理如下：

| 启动结果 | 处理 |
| --- | --- |
| 启动成功 | 立即执行正常停止直播流程 |
| 启动失败 | 清理启动状态和已打开资源 |
| 启动超时 | 清理启动状态和已打开资源 |
| 到达安全点 | 执行延迟停止并释放直播资源 |

这里的关键点是：即使后台或用户只发了一次停止，设备也要靠内部记录的延迟停止请求完成后续停止。

## 八、唤醒模式下的补充说明

唤醒模式下，直播停止不等于马上对设备下电。

真正停止直播后，还要继续结合唤醒模式的保活和定时拍照窗口判断。

| 条件 | 处理 |
| --- | --- |
| 仍在保活时间内 | 按保活策略继续供电 |
| 处于可见光定时拍照前开启窗口 | 云台和可见光不下电 |
| 处于红外定时拍照前开启窗口 | 云台、可见光和红外不下电 |
| 不在保活时间，也不在开启窗口，设备空闲 | 按唤醒模式策略下电 |

因此，本方案只处理直播启动和停止的业务状态，不改变原有唤醒模式保活策略。

## 九、正常模式下的补充说明

正常模式下通常不需要频繁上下电，但仍然需要处理启动中停止和新拉流复用。

| 条件 | 处理 |
| --- | --- |
| 启动中收到停止 | 立即响应，内部延迟停止 |
| 后续有新 SSRC | 取消旧停止，复用当前启动 |
| 后续没有新 SSRC | 启动收口后停止直播资源 |
| 直播停止后 | 按正常模式原有策略释放直播和编码资源 |

## 十、建议修改方向

后续代码建议优先在 `MainActivity` 的直播入口层处理，不建议第一步就改底层设备打开逻辑。

| 顺序 | 修改方向 | 说明 |
| --- | --- | --- |
| 1 | 增加每通道直播启动状态 | 记录是否启动中、当前 SSRC、是否延迟停止 |
| 2 | 修改开始直播逻辑 | 启动中收到新拉流时，复用当前启动流程 |
| 3 | 修改停止直播逻辑 | 启动中收到停止时，立即响应，但只记录延迟停止 |
| 4 | 修改打开成功回调 | 启动成功后检查是否需要停止，或是否服务新 SSRC |
| 5 | 修改打开失败回调 | 清理启动状态，避免状态残留 |
| 6 | 增加日志 | 记录开始、停止、新 SSRC 接管、延迟停止执行等关键点 |

## 十一、建议验证场景

| 场景 | 预期结果 |
| --- | --- |
| 红外启动中手动停止，后续没有新拉流 | 停止命令立即响应，启动收口后自动停止 |
| 红外启动中手动停止，随后又拉红外 | 新拉流接管当前启动，旧停止不再生效 |
| 可见光启动中手动停止，后续没有新拉流 | 停止命令立即响应，启动收口后自动停止 |
| 可见光启动中手动停止，随后新 SSRC 拉流 | 新 SSRC 复用当前启动并正常出流 |
| 启动成功后再停止 | 走原有正常停止流程 |
| 启动失败或超时 | 清理启动状态，不残留延迟停止 |

## 十二、总结

这个问题的解决重点是建立直播启动状态和延迟停止机制。

最终效果应该是：

停止指令立即响应。

启动流程不被强制打断。

停止请求不会丢失。

新 SSRC 拉流可以接管当前启动流程。

没有新拉流时，设备会在启动收口后自动停止。
