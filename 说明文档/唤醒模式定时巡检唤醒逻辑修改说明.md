# 定时巡检唤醒与通道扩展逻辑修改说明

## 一、问题现象

当前执行定时巡检时，代码直接进入巡检执行流程，没有像直播拉流、拍照、短视频那样先唤醒并打开设备。

在基础模式、唤醒模式下，如果云台、可见光机芯、红外机芯已经下电，定时巡检到点后可能出现以下问题：

| 问题 | 说明 |
| --- | --- |
| 云台未上电 | 巡检动作调用预置位时，云台无法正常转动 |
| 可见光机芯未上电 | 巡检前设置图片参数、OSD、视频参数等操作可能失败 |
| 网口未打开 | 需要访问机芯或云台时，通信可能失败 |
| 巡检直接执行 | 没有等待设备启动完成，导致巡检动作无效 |
| 通道二无法定时巡检 | 当前定时巡检固定通道一，通道二巡检计划不会被独立执行 |

新的需求是：

| 需求 | 说明 |
| --- | --- |
| 全功能模式支持通道二巡检 | 通道二红外也需要支持定时巡检 |
| 基础模式支持巡检 | 基础模式下执行巡检时，也需要像拉流一样先唤醒设备 |
| 逻辑和拉流一致 | 巡检执行前先上电、打开 Share、等待设备 ready，再执行巡检 |
| 通道一和通道二分别执行 | 巡检时间表、闹钟、执行流程都需要带通道号 |

## 二、当前代码流程

当前定时巡检主要流程如下：

| 步骤 | 当前处理 |
| --- | --- |
| 1 | `initCheckLineTask()` 根据巡检时间表设置 `ACTION_CHECK_LINE` 闹钟 |
| 2 | 闹钟触发后进入 `doCheckLineAction()` |
| 3 | `doCheckLineAction()` 固定调用 `runCheckLine(1, item.group - 1, item.count)` |
| 4 | `runCheckLine()` 读取巡检组配置 |
| 5 | 设置图片参数、OSD、视频参数、录像策略 |
| 6 | 打开 Share |
| 7 | 等待 3 秒 |
| 8 | 执行 `dev.startCheckLine(item)` |
| 9 | 巡检结束后回到录像预置位 |

当前流程有两个限制：

| 限制 | 说明 |
| --- | --- |
| 没有唤醒入口 | 基础模式、唤醒模式下不会像拉流一样先唤醒设备 |
| 固定通道一 | `initCheckLineTask()` 固定读取通道一时间表，`doCheckLineAction()` 固定执行通道一 |

## 三、拉流逻辑中的唤醒处理

直播拉流在低功耗相关模式下会先执行唤醒逻辑。

| 位置 | 处理 |
| --- | --- |
| `startLiveVideo()` | 判断当前模式和目标通道 |
| `markWakeupActivity()` | 刷新唤醒模式业务状态和保活计时 |
| `wakeupDevice()` | 根据设备类型执行上电、打开 Share、USB 权限等处理 |
| `dev.open()` | 等待设备打开完成后再开始直播 |

可见光拉流大致流程：

| 步骤 | 处理 |
| --- | --- |
| 1 | 收到可见光拉流请求 |
| 2 | 唤醒模式下标记业务保活 |
| 3 | 云台和可见光机芯上电 |
| 4 | 打开 Share 和网络 |
| 5 | 等待设备 ready |
| 6 | 设备打开成功后开始推流 |

定时巡检应当参考这个流程。不同模式下都要保证“设备能执行巡检”：

| 模式 | 巡检前处理 |
| --- | --- |
| 全功能模式 | 通道一、通道二都允许巡检；如果设备已经上电，直接打开或复用设备；如果设备未 ready，仍要等待 ready |
| 基础模式 | 按拉流逻辑先唤醒对应负载，再等待设备 ready |
| 唤醒模式 | 按拉流逻辑先唤醒对应负载，并刷新唤醒业务保活 |
| 休眠模式 | 不执行巡检，或直接返回失败/跳过 |

## 四、定时巡检目标逻辑

定时巡检应该支持通道一和通道二，并且在基础模式、唤醒模式下先唤醒设备，再执行巡检。

| 阶段 | 目标处理 |
| --- | --- |
| 巡检闹钟触发 | 进入 `doCheckLineAction()` |
| 执行巡检前 | 判断当前模式和巡检通道 |
| 唤醒设备 | 按拉流逻辑唤醒云台和对应机芯 |
| 等待设备打开 | 等待云台、机芯、网络 ready |
| 执行巡检 | 调用巡检组中的特殊预置位 |
| 巡检结束 | 按当前模式的业务结束规则判断下电 |

## 五、通道一可见光定时巡检逻辑

通道一对应可见光巡检。

| 步骤 | 处理 |
| --- | --- |
| 1 | 定时巡检闹钟触发 |
| 2 | 获取通道一设备 |
| 3 | 如果是唤醒模式，调用 `markWakeupActivity(dev, "唤醒模式下可见光定时巡检")` |
| 4 | 如果是基础模式或唤醒模式，调用 `wakeupDevice(dev, "可见光定时巡检")` |
| 5 | 等待设备打开或 ready |
| 6 | 设置图片参数、OSD、视频参数、录像策略 |
| 7 | 打开 Share |
| 8 | 执行巡检组 |
| 9 | 巡检结束后回到录像预置位 |
| 10 | 按当前模式的业务结束规则处理 |

## 六、通道二红外定时巡检逻辑

通道二对应红外巡检。红外巡检不能只开启红外机芯，还需要云台和可见光机芯参与 ready 判断，因此唤醒对象要和红外拉流一致。

| 步骤 | 处理 |
| --- | --- |
| 1 | 定时巡检闹钟触发 |
| 2 | 获取通道二设备 |
| 3 | 如果是唤醒模式，调用 `markWakeupActivity(dev, "唤醒模式下红外定时巡检")` |
| 4 | 如果是基础模式或唤醒模式，调用 `wakeupDevice(dev, "红外定时巡检")` |
| 5 | 唤醒云台、可见光机芯和红外机芯 |
| 6 | 等待红外识别成功，同时等待云台或可见光 ready |
| 7 | 设置红外相关参数 |
| 8 | 执行巡检组 |
| 9 | 巡检结束后停止巡检 |
| 10 | 按当前模式的业务结束规则处理 |

## 七、巡检闹钟扩展逻辑

当前巡检闹钟固定通道一，需要扩展为按通道创建和取消。

| 修改点 | 说明 |
| --- | --- |
| 闹钟 requestCode | 使用独立基准 + 通道偏移 + 策略序号，避免通道一和通道二互相覆盖 |
| 闹钟 Intent | 增加 `channel` 和 `index` |
| 初始化巡检任务 | 分别初始化通道一、通道二的巡检闹钟 |
| 取消巡检任务 | 按通道精确取消旧巡检闹钟 |
| 闹钟触发 | `doCheckLineAction()` 从 Intent 读取通道号 |
| 执行巡检 | 调用 `runCheckLine(channel, item.group - 1, item.count)` |

建议新增类似拍照预热闹钟的 requestCode 计算方式：

| 项目 | 说明 |
| --- | --- |
| `RC_CHECK_LINE_BASE` | 巡检闹钟独立基准 |
| `RC_CHECK_LINE_STRIDE` | 每个通道的 requestCode 间隔 |
| `checkLineRequestCode(channel, index)` | 用于生成通道独立、策略独立的 requestCode |

## 八、建议代码修改点

### 1. 在巡检执行前增加唤醒入口

建议在 `runCheckLine()` 开始处增加低功耗模式处理。

目标逻辑：

| 条件 | 处理 |
| --- | --- |
| 当前是全功能模式 | 支持通道一、通道二巡检，设备未 ready 时仍等待 ready |
| 当前是基础模式，且设备是 DVR | 调用和拉流一致的唤醒流程 |
| 当前是唤醒模式，且设备是 DVR | 调用和拉流一致的唤醒流程，并刷新保活 |
| 当前是休眠模式 | 不执行巡检 |
| 当前是可见光 | 唤醒云台和可见光机芯 |
| 当前是红外 | 唤醒云台、可见光机芯和红外机芯 |

### 2. 保活标记

巡检属于业务任务。在唤醒模式下，应当刷新保活状态。

| 处理 | 说明 |
| --- | --- |
| `markWakeupActivity(dev, "唤醒模式下可见光定时巡检")` | 标记通道一巡检业务 |
| `markWakeupActivity(dev, "唤醒模式下红外定时巡检")` | 标记通道二巡检业务 |
| 保活时间 | 先沿用普通业务保活时间 |
| 巡检结束 | 走统一下电判断 |

### 3. 设备唤醒

基础模式、唤醒模式下，巡检执行前调用：

| 设备类型 | 处理 |
| --- | --- |
| 通道一可见光 | `wakeupDevice(dev, "可见光定时巡检")` |
| 通道二红外 | `wakeupDevice(dev, "红外定时巡检")` |

这样可以复用拉流中的上电和 Share 打开逻辑。

### 4. 等待设备 ready

定时巡检不能只上电后立即执行预置位，应该等待设备 ready。

建议优先复用设备已有的 `open()` 逻辑，原因如下：

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 只调用 `wakeupDevice()` 后 sleep 固定时间 | 修改少 | 时间不稳定，设备未 ready 时仍可能失败 |
| 调用 `dev.open()` 等待设备 ready | 和拉流、拍照一致 | 需要避免重复打开或影响正在执行的任务 |

建议采用第二种：唤醒后调用 `dev.open()`，打开成功再执行巡检。

## 九、巡检结束后的下电逻辑

巡检结束后不应立即无条件下电，应按当前模式走统一判断。

| 情况 | 处理 |
| --- | --- |
| 巡检结束后设备空闲，且不在拍照提前窗口 | 允许下电 |
| 巡检结束后仍有直播、拍照、短视频、回放等任务 | 不下电，等待下一轮判断 |
| 巡检结束后处于可见光拍照前 3 分钟窗口 | 不下电 |
| 巡检结束后处于红外拍照前 10 分钟窗口 | 不下电 |

当前可以复用 `sleepDevice(channel, "定时巡检完成")` 或 `doSleep("定时巡检完成", load)` 的统一判断。

## 十、建议后的完整流程

| 步骤 | 处理 |
| --- | --- |
| 1 | 巡检闹钟触发 |
| 2 | 从 Intent 读取通道号和巡检策略序号 |
| 3 | 获取通道设备 |
| 4 | 如果是唤醒模式，标记巡检业务保活 |
| 5 | 如果是基础模式或唤醒模式，按拉流逻辑唤醒设备 |
| 6 | 等待设备打开成功 |
| 7 | 设置图片参数、OSD、视频参数、录像策略 |
| 8 | 打开 Share |
| 9 | 执行巡检组 |
| 10 | 巡检结束，停止巡检 |
| 11 | 工作时间内回录像预置位 |
| 12 | 按当前模式统一下电判断处理 |
| 13 | 初始化该通道下一次巡检闹钟 |

## 十一、需要注意的点

| 注意项 | 说明 |
| --- | --- |
| 全功能模式也要支持通道二 | 不能只扩展唤醒模式，通道二红外在全功能模式下也要能定时巡检 |
| 基础模式需要唤醒 | 基础模式下执行巡检前也要像拉流一样先唤醒设备 |
| 不要影响 MIPI | 当前巡检不是 MIPI 任务，不需要加入 MIPI 上下电 |
| 通道一和通道二闹钟要独立 | 避免两个通道的巡检闹钟互相覆盖或取消错 |
| 避免重复打开设备 | 如果设备已经打开或正在忙，需要按现有设备状态判断 |
| 结束后不要直接断电 | 必须经过统一下电判断，避免影响后续任务或提前开启窗口 |
| 休眠模式不执行巡检 | 休眠模式应直接跳过或返回失败，避免误上电 |
