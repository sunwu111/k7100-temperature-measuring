# 高德红外直播拉流与短视频录制并行设计方案

## 1. 背景

当前高德红外 `GUIDEDev` 的直播和短视频录制都依赖同一个 USB 红外机芯预览源。

后台正在拉高德红外直播时，如果再下发短视频录制命令，当前调度逻辑会先停止直播，再启动短视频录制。录制完成后不会自动恢复之前的直播，需要后台重新拉流。

因此当前表现是：

1. 后台开始红外直播。
2. 设备打开高德红外机芯预览。
3. 设备不断读取红外预览帧，编码后推给后台。
4. 后台下发短视频录制。
5. 程序先停止红外直播。
6. 再重新启动高德红外预览用于录制。
7. 录制完成后释放预览、编码器和文件写入资源。
8. 原直播不再继续。

本质原因是直播和录制共用了同一个预览生命周期和编码链路，当前设计把它们当成互斥任务。

### 1.1 是否支持并行处理
高德红外预览或编码能力是否支持并行处理

## 2. 目标

本方案目标是让高德红外支持直播和短视频录制并行：

1. 后台正在看红外直播时，可以下发短视频录制。
2. 录制短视频时不能主动停止正在进行的直播。
3. 录制完成后只结束录制状态，不结束直播状态。
4. 直播继续保持，直到后台明确下发停止直播。


## 3. 当前逻辑说明

### 3.1 高德红外直播逻辑

高德红外直播不是 RTSP 拉流。

它的直播数据来自本地 USB 红外机芯预览帧。

当前直播流程是：

1. 后台下发开始直播。
2. 程序进入高德红外直播启动流程。
3. 设置设备为直播状态。
4. 根据码流配置确定输出分辨率。
5. 初始化视频编码器。
6. 创建 RTP 打包器，并使用后台下发的 SSRC。
7. 启动高德红外预览。
8. 循环等待红外预览帧。
9. 把预览帧转换成 Bitmap。
10. 绘制水印、温度区域、调色板、热点追踪、防灼伤保护等内容。
11. 将处理后的 Bitmap 编码成视频帧。
12. 将编码数据封装成 RTP 后推给后台。

直播停止时，会停止红外预览、关闭机芯相关资源，并清除直播状态。

### 3.2 高德红外短视频录制逻辑

高德红外短视频录制同样来自本地 USB 红外机芯预览帧。

当前录制流程是：

1. 后台下发短视频录制。
2. 程序检查设备是否正在录制。
3. 如果正在录制，则拒绝新的录制任务。
4. 如果正在直播，当前调度层会先停止直播。
5. 启动高德红外录制流程。
6. 设置设备为录制状态。
7. 创建 MP4 文件写入器。
8. 初始化视频编码器。
9. 如启用音频，则初始化音频采集和音频编码。
10. 启动高德红外预览。
11. 循环等待红外预览帧。
12. 把预览帧转换成 Bitmap。
13. 绘制水印、调色板、热点追踪、防灼伤保护等内容。
14. 将处理后的 Bitmap 编码成视频帧。
15. 将编码数据写入 MP4 文件。
16. 到达录制时长后停止录制。
17. 停止预览、关闭机芯、释放编码器、释放 MP4 写入器。
18. 生成短视频文件并进入上传流程。

### 3.3 当前互斥点

当前直播和录制的互斥点主要有三个：

1. 调度层在录制前会主动停止直播。
2. 直播和录制都会启动和停止同一个高德红外预览源。
3. 直播和录制都会使用视频编码器，但当前没有区分“直播编码”和“录制编码”的资源边界。

所以高德红外当前不是并行模式，而是资源切换模式。

## 4. 并行处理核心思路

高德红外并行方案不能按照可见光机芯的“两路 RTSP”方法。

高德红外的数据源是本地 USB 预览帧，同一时刻主要只有一个红外预览源。

因此高德红外更适合采用“一路预览源，分发给两个业务消费者”的方案。

核心思路是：

1. 高德红外机芯只启动一次预览。
2. 预览线程只负责持续产出红外帧。
3. 直播业务从同一帧源取帧，处理后编码并推给后台。
4. 录制业务从同一帧源取帧，处理后编码并写入 MP4。
5. 直播和录制分别维护自己的状态。
6. 直播停止只停止直播消费者。
7. 录制停止只停止录制消费者。
8. 只有直播和录制都结束后，才允许停止红外预览和关闭机芯。

## 5. 推荐业务模型

### 5.1 预览源

预览源是高德红外机芯输出的原始图像帧。

它应该独立于直播和录制存在。

只要当前存在直播或录制任意一个业务，就保持预览源开启。

只有所有业务都结束后，才关闭预览源。

### 5.2 直播消费者

直播消费者负责把预览帧变成后台实时流。

它的职责包括：

1. 读取最新红外帧。
2. 绘制直播需要的水印和温度区域。
3. 绘制直播需要的调色板、热点追踪、防灼伤保护。
4. 编码成直播视频帧。
5. 封装成 RTP。
6. 按后台下发的 SSRC 推给后台。

直播消费者的生命周期由后台开始直播和停止直播命令控制。

### 5.3 录制消费者

录制消费者负责把预览帧变成 MP4 文件。

它的职责包括：

1. 读取最新红外帧。
2. 绘制录制文件需要的水印和红外附加内容。
3. 编码成录制视频帧。
4. 写入 MP4 文件。
5. 到达录制时长后停止。
6. 完成文件收口和上传。

录制消费者的生命周期由短视频录制命令和录制时长控制。


## 6. 状态关系

高德红外允许以下状态同时存在：

1. 预览源已开启。
2. 设备处于直播状态。
3. 设备处于录制状态。

状态含义如下：

1. 预览源状态表示高德红外机芯正在输出图像帧。
2. 直播状态表示正在向后台推实时流。
3. 录制状态表示正在生成短视频文件。

状态之间的关系：

1. 直播启动时，如果预览源未开启，则开启预览源。
2. 录制启动时，如果预览源未开启，则开启预览源。
3. 直播停止时，只关闭直播消费者。
4. 录制停止时，只关闭录制消费者。
5. 直播和录制都停止后，才关闭预览源和机芯。

## 7. 推荐流程

### 7.1 直播中启动短视频

推荐流程：

1. 后台下发开始直播。
2. 高德红外开启预览源。
3. 直播消费者开始读取预览帧并推流。
4. 后台下发短视频录制。
5. 程序发现当前正在直播，但不停止直播。
6. 程序检查是否已经在录制。
7. 如果没有录制，则启动录制消费者。
8. 直播消费者继续推流。
9. 录制消费者同时写 MP4。
10. 录制时间到后，录制消费者停止。
11. MP4 文件完成收口并上传。
12. 直播继续保持。
13. 后台下发停止直播后，直播消费者停止。
14. 如果此时没有录制任务，再关闭预览源和机芯。

### 7.2 录制中启动直播

推荐流程：

1. 后台先下发短视频录制。
2. 高德红外开启预览源。
3. 录制消费者开始写 MP4。
4. 后台再下发开始直播。
5. 程序发现当前正在录制，但不拒绝直播。
6. 启动直播消费者。
7. 录制继续写 MP4。
8. 直播开始推给后台。

这个流程可以保证录制和直播谁先来都能共用预览源。

### 7.3 重复录制

如果录制过程中再次收到短视频录制命令，保持当前策略：

1. 不允许同一通道同时有两个录制消费者。
2. 新录制任务等待。
3. 当前录制继续。
4. 当前直播不受影响。

### 7.4 停止直播

如果录制过程中后台下发停止直播：

1. 停止直播消费者。
2. 关闭后台推流连接。
3. 清除直播状态。
4. 不停止录制消费者。
5. 如果录制仍在进行，预览源继续保持。

### 7.5 录制完成

如果直播过程中录制完成：

1. 停止录制消费者。
2. 释放录制文件写入资源。
3. 生成文件并上传。
4. 清除录制状态。
5. 不停止直播消费者。
6. 不关闭预览源。

## 8. 资源隔离原则

高德红外并行设计需要遵守以下原则：

1. 红外预览源可以共享。
2. 直播输出和录制输出不能混用。
3. 直播编码资源和录制编码资源应独立。
4. 直播停止不能释放录制资源。
5. 录制停止不能释放直播资源。
6. 预览源关闭必须等待直播和录制都结束。
7. 红外机芯关闭必须等待所有消费者都结束。
8. 文件写入异常不能影响直播推流。
9. 后台推流异常不能影响短视频录制。


## 9. 唤醒模式处理

唤醒模式下，高德红外直播或录制都会让红外机芯和相关负载保持上电。

并行后需要遵守：

1. 只要直播还在，录制完成不能触发红外下电。
2. 只要录制还在，直播停止不能触发红外下电。
3. 只有直播和录制都结束后，才进入原有后续任务判断。
4. 如果后续存在红外拍照任务，应继续保持预热或保活。
5. 如果没有后续任务，再按唤醒模式规则关闭红外和相关负载。

## 10. 可能存在的问题

### 10.1 编码资源压力

并行后可能需要两套编码输出：

1. 一套用于直播 RTP。
2. 一套用于录制 MP4。

可能出现帧率下降、编码延迟、直播卡顿或录制掉帧。

### 10.2 Bitmap 处理开销

直播和录制都需要绘制水印、温度区域、调色板和防护效果。

如果两边各自完整处理一遍，CPU，内存压力会明显增加。
