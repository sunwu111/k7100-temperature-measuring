# 双路 MIPI 同时拉流设计文档

## 一、背景

当前 MIPI 摄像头业务需要支持两路 MIPI 同时工作。原先单路 MIPI 逻辑更偏向“当前通道打开、当前通道出帧、当前通道释放”，当两个 MIPI 通道都需要拉流时，容易出现以下问题：

- 只打开当前通道，另一通道无法提前建立相机资源。
- 两路相机打开过程互相抢占，导致某一路打开失败或状态错乱。
- 停止一路拉流时，如果直接释放相机资源，会影响另一条仍在拉流的 MIPI。
- 拉流过程中拍照如果直接使用视频帧截图，图片清晰度和噪声控制不如静态拍照。

因此，当前设计把 MIPI 逻辑从“单通道独立打开”调整为“双路相机统一协调打开、各自独立出帧和推流”。

## 二、设计目标

本设计要满足以下目标：

1. 任意一路 MIPI 拉流到来时，能够同时准备两路 MIPI 相机资源。
2. 两路 MIPI 能够分别出帧、分别编码、分别推送，不互相合流。
3. 第二路拉流到来时复用已经打开的双路相机资源，不重复打开底层相机。
4. 停止一路拉流时，只停止该路编码和推流，不影响另一条仍在工作的 MIPI。
5. 只有两路都没有直播、拍照、录像任务时，才释放两个相机资源。

## 三、整体架构

双路 MIPI 的整体架构可以理解为四层：

| 层级 | 作用 |
| --- | --- |
| 通道对象层 | 每个 MIPI 通道各自有一个摄像头对象，分别对应一路物理 MIPI。 |
| 双路协调层 | 保存两路相机对象引用，统一判断是否正在打开、是否已经打开、是否可以释放。 |
| 相机会话层 | 每路相机各自建立视频输出会话，负责持续输出帧。 |
| 编码推流层 | 每路根据自己的拉流请求独立编码并发送到对应后台会话。 |

这里的“双路同时拉流”不是把两个 MIPI 合成一个视频流，而是：

- 相机 A 独立打开、独立出帧、独立编码、独立推流。
- 相机 B 独立打开、独立出帧、独立编码、独立推流。
- 两路只在“打开和释放相机资源”时进行统一协调。

## 四、对象登记机制

每个 MIPI 通道对象创建时，会根据物理摄像头编号登记为两路之一：

| 物理关系 | 逻辑含义 |
| --- | --- |
| 相机 A | 第一条 MIPI 相机链路 |
| 相机 B | 第二条 MIPI 相机链路 |

登记完成后，任意一路收到拉流请求，都可以找到另一条 MIPI 对象。

这样做的目的，是让任意一路业务都能触发“双路资源准备”，避免只打开当前一路，导致后续第二路拉流时再去抢资源。

## 五、双路打开流程

当后台或本地预览发起任意一路 MIPI 拉流时，系统先进入双路打开协调流程。

打开流程逻辑如下：

1. 收到某一路 MIPI 拉流请求。
2. 进入全局互斥判断，防止两个拉流请求同时触发重复打开。
3. 判断两路 MIPI 对象是否都已经登记。
4. 如果缺少任意一路对象，则打开失败。
5. 如果双路正在打开或已经打开，则直接认为打开成功。
6. 如果还没有打开，则标记为“双路正在打开”。
7. 依次打开两路底层相机。
8. 两路底层相机都打开成功后，再分别创建两路视频会话。
9. 两路会话都成功后，标记“双路已打开”。

这个流程保证了：

- 双路打开只会有一个主流程在执行。
- 第二个同时到来的打开请求不会重复打开底层相机。
- 两路相机资源要么都准备成功，要么整体认为打开失败。

## 六、视频会话设计

每一路 MIPI 都会建立自己的视频会话。

视频会话中包含：

| 输出 | 用途 |
| --- | --- |
| 视频帧输出 | 用于直播拉流和短视频录制的帧处理。 |
| 静态拍照输出 | 用于拉流过程中高清 JPEG 抓拍。 |

视频帧输出负责持续产生图像帧，后续进入水印、编码、推流流程。

静态拍照输出平时不影响拉流，只在收到拍照请求时触发一次静态抓拍。这样可以避免拉流中拍照退化成视频帧截图。

## 七、拉流执行逻辑

双路相机资源建立后，每一路拉流仍然是独立执行的。

### 7.1 单路拉流

如果只有相机 A 被要求拉流：

1. 双路底层相机仍然都会被准备好。
2. 相机 A 启用编码和推流。
3. 相机 B 虽然已经打开并具备出帧能力，但不一定启用编码发送。

如果只有相机 B 被要求拉流，逻辑相同。

### 7.2 双路同时拉流

如果 A、B 两路都收到拉流请求：

1. 双路相机资源已经存在。
2. A 路启用自己的编码器和推流目标。
3. B 路启用自己的编码器和推流目标。
4. 两路各自发送到对应的后台会话。


## 八、拉流中拍照设计

拉流中拍照有两种处理路径：

| 场景 | 处理方式 |
| --- | --- |
| 当前会话支持静态 JPEG 输出 | 触发静态 JPEG 抓拍，保存高清图片。 |
| 当前会话不支持静态 JPEG 输出 | 退回使用当前视频帧截图，保证业务不中断。 |

当前设计优先使用静态 JPEG 输出。这样做的原因是：

- JPEG 静态输出更接近真正拍照链路。
- 图片不再完全依赖 YUV 视频预览帧。
- 暗光、噪声和清晰度表现通常会比视频截图更好。

## 九、停止与释放逻辑

停止一路 MIPI 拉流时，不会立即释放两个相机。

停止流程如下：

1. 收到某一路停止拉流请求。
2. 关闭该路编码和推流状态。
3. 清理该路直播标记。
4. 判断两路是否还有直播、拍照、录像任务。
5. 如果另一条路仍在工作，则保留两个相机资源。
6. 如果两路都已经空闲，则释放两个相机、视频会话和图像输出。

这个设计可以避免：

- 停止 A 路时误关 B 路。
- 拍照过程中被停止拉流释放资源。
- 录像过程中被另一条路停止动作影响。
