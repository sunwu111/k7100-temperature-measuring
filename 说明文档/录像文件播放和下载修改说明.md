# 录像文件播放和录像文件下载修改说明

本文记录录像文件播放和录像文件下载相关功能的修改思路、修改过程、涉及文件和当前代码逻辑，方便后续维护人员理解为什么这样实现。

## 一、涉及的主要文件

+ `app/src/main/java/hikvision/zhanyun/com/hikvision/SPGProtocol.java`
  + 负责解析服务器下发的协议指令。
  + 录像文件数量查询：`98H`
  + 录像文件列表查询：`99H`
  + 录像文件回放：`9AH`
  + 录像回放控制：`9BH`
  + 录像回放停止：`9CH`
  + 录像文件下载：`9DH`
  + 录像文件下载取消：`9EH`

+ `app/src/main/java/hikvision/zhanyun/com/hikvision/MainActivity.java`
  + 实现 `SPGPCallback` 中的回放、控制、停止回调。
  + 负责唤醒模式下的云台/红外上电、保活、下电。
  + 负责创建服务器接收流的 `SocketClient`。

+ `app/src/main/java/hikvision/zhanyun/com/hikvision/device/HuanYuDevice.java`
  + 负责环宇设备的 RTSP 回放拉流。
  + 将设备侧 RTSP 回放数据转换为服务器需要的 RTP 数据。
  + 通过 `playbackSession` 过滤旧连接残留数据。

+ `app/src/main/java/hikvision/zhanyun/com/hikvision/utils/VideoFiles.java`
  + 唤醒模式下，录像文件查询可读取缓存。
  + 用于避免设备刚进入唤醒模式时，为简单查询频繁打开机芯。

## 三、问题背景

### 1. 点击播放后立即停止

之前验证录像回放时，出现过“点击播放后直接停止”的现象。这个问题的主要风险点有两个：

+ 旧回放连接没有完全释放，新回放已经启动。
+ 旧 RTSP 连接仍可能继续回调数据，导致新旧回放流互相干扰。

如果停止回放时同步等待底层 RTSP 断开，新的回放请求可能被阻塞；如果不处理旧连接回调，旧连接残留数据又可能继续往服务器发送。

因此后续修改的重点是：停止时先让旧 session 失效，并且立刻清状态，不让阻塞的 RTSP 断开过程影响新的回放。

### 2. 录像文件下载和录像回放使用同一套协议参数

从日志和后台操作看，服务器下发录像文件下载时，数据域结构与录像回放基本一致：

+ 通道号
+ 传输方式
+ 服务器 IP
+ 服务器端口
+ 起止时间
+ SSRC

也就是说，所谓“录像文件下载”并不是设备先在本地生成一个文件再上传，而是设备按指定时间段把录像流推送到服务器，由服务器接收保存。

所以最终实现上没有单独写一套下载逻辑，而是让 `9DH` 走 `9AH` 一样的回放流流程，只区分响应控制字和日志名称。

## 四、录像回放的当前流程

### 1. 协议入口

在 `SPGProtocol.java` 中，收到 `9AH` 后进入：

```java
private void doPlayback(byte[] raw) {
    doPlaybackStream(raw, ORDER_9AH, "启动回放");
}
```

这里不直接写解析逻辑，而是调用通用方法 `doPlaybackStream(...)`。

### 2. 统一解析回放/下载参数

`doPlaybackStream(...)` 根据报文长度判断新旧协议：

+ 新协议：长度较短，IP 是 4 字节 IPv4。
+ 旧协议：长度较长，IP 是 16 字节字符串。

解析出来的核心参数包括：

+ `channel`：通道号。
+ `UDP`：是否 UDP 传输。
+ `ip`：服务器接收流的地址。
+ `port`：服务器接收流的端口。
+ `startVideoTime`：录像开始时间。
+ `stopVideoTime`：录像结束时间。
+ `ssrc`：本次回放/下载流的 SSRC。

解析完成后调用：

```java
byte ret = listenerCallBack.playbackFile(
        raw[10],
        raw[11] == 0,
        startVideoTime,
        stopVideoTime,
        ip,
        port,
        ssrc);
```

也就是说，协议层只负责解析和响应，真正打开设备、拉流、推流的逻辑在 `MainActivity.playbackFile(...)` 中。

### 3. 响应服务器

回放响应的数据域为：

+ 通道号
+ 结果码
+ 保留字节
+ SSRC

回放时发送：

```java
sendPack(ORDER_9AH, null, dataDomain);
```

下载时发送：

```java
sendPack(ORDER_9DH, null, dataDomain);
```

这就是为什么要把响应控制字作为参数传入 `doPlaybackStream(...)`。

## 五、录像下载的当前流程

### 1. 协议入口

在 `SPGProtocol.java` 中，收到 `9DH` 后进入：

```java
private void doVideoDownload(byte[] mReceiveData){
    doPlaybackStream(mReceiveData, ORDER_9DH, "启动录像下载");
}
```

这里明确复用了回放流处理逻辑。

### 2. 为什么 9DH 复用 9AH

原因是后台下发的下载指令与回放指令在数据结构和行为上是一致的：

+ 都指定一个录像时间段。
+ 都指定服务器 IP 和端口。
+ 都要求设备把录像流发给服务器。
+ 都需要 SSRC 标识这一路流。

两者的区别主要是服务器侧语义不同：

+ `9AH`：服务器认为这是“播放”。
+ `9DH`：服务器认为这是“下载保存”。

对设备端来说，都是“按时间段取录像并推送到服务器”。因此复用同一条链路可以避免两个解析函数长期不一致。

### 3. 下载取消

收到 `9EH` 时进入：

```java
private void doVideoDownloadCancel(byte[] mReceiveData)
```

处理步骤：

+ 校验接收数据长度。
+ 解析通道号。
+ 解析 SSRC。
+ 调用 `listenerCallBack.stopPlayCallBack(channel, parsedSSRC)`。
+ 用 `ORDER_9EH` 回复服务器。

下载取消本质上也是停止当前录像流，所以复用了回放停止回调。

## 六、MainActivity 中的回放处理

### 1. 唤醒模式下先上电并保活

`playbackFile(...)` 中如果当前是唤醒模式，会先调用：

```java
markWakeupActivity(dev, dev.isUSB() ? "唤醒模式下播放红外录像文件" : "唤醒模式下播放可见光录像文件");
isWakeupVideoPlaybackMode = true;
```

作用是：

+ 可见光通道：开启云台。
+ 红外通道：开启云台和红外。
+ 刷新唤醒模式保活定时器。
+ 防止回放/下载过程中被空闲下电。

### 2. 打开设备前先处理旧回放

如果设备已经处于回放状态，会先停止旧回放：

```java
if (dev.isPlaybacking()) {
    dev.playbackStop();
    Thread.sleep(200);
}
```

这样做是为了避免同一通道同时存在两路回放。

### 3. 重新创建服务器推流连接

如果之前已经存在 `streamClient`，会先关闭：

```java
if (dev.streamClient != null) {
    dev.streamClient.close();
}
```

然后根据服务器下发的 IP、端口和传输方式创建新的 `SocketClient`。

这里有一个特殊处理：

```java
if ("183.47.15.148".equals(ip)) {
    dev.streamClient = new SocketClient(ip, port, false);
} else {
    dev.streamClient = new SocketClient(ip, port, UDP);
}
```

这表示特定服务器地址不按下发的 UDP 标志走，而是强制使用 `false` 对应的传输方式。该逻辑是已有兼容处理，当前录像下载复用了这条回放链路，因此也会继承这个行为。

### 4. 打开机芯后启动回放

设备打开成功后调用：

```java
dev.playbackStart(startTime, stopTime, ssrc)
```

也就是说，`MainActivity` 负责连接服务器和打开设备，真正从设备拉取录像流是在具体 `Device` 子类中完成。

## 七、HuanYuDevice 中的回放稳定性处理

### 1. 使用 playbackSession 过滤旧连接

`HuanYuDevice` 中有：

```java
private final AtomicLong playbackSession = new AtomicLong(0);
```

每次 `playbackStart(...)` 都会：

```java
final long currentSession = playbackSession.incrementAndGet();
```

RTSP 回调收到数据时会判断：

```java
if (currentSession != playbackSession.get()) {
    return;
}
```

如果当前回调属于旧 session，就直接丢弃，不再转发给服务器。

这解决的是“旧回放连接还在吐数据”的问题。

### 2. 回放 RTP 包替换 SSRC

服务器下发的 SSRC 要出现在最终发送给服务器的 RTP 数据里，所以回放数据回调中会写入：

```java
data[8] = (byte) (ssrc >> 24);
data[9] = (byte) (ssrc >> 16);
data[10] = (byte) (ssrc >> 8);
data[11] = (byte) (ssrc);
```

这样服务器能按本次指令的 SSRC 识别这一路回放/下载流。

### 3. 停止回放时先清状态，再异步停止 RTSP

当前停止逻辑是：

+ 先保存当前 `rtspPlaybackClient`。
+ 将 `rtspPlaybackClient` 置空。
+ 递增 `playbackSession`，让旧回调全部失效。
+ 清空 `ssrcPlayback`。
+ 清除 `PLAYBACKING` 状态。
+ 新开线程执行真正的 `stoppingClient.stop()`。

这样做的原因是 RTSP `TEARDOWN/stop` 可能阻塞，如果同步执行，会影响后续新回放或下载立即启动。

## 八、录像文件查询与唤醒模式缓存

录像播放/下载前通常会先查询文件数量和文件列表。

唤醒模式下相关入口：

+ `fileFiles(...)`
+ `findVideoFileList(...)`

当前逻辑是：

+ 如果是唤醒模式，会调用 `markWakeupActivity(...)`，保证云台/红外上电并刷新保活。
+ 如果还没有进入真正的回放流程，允许读取 `VideoFiles` 缓存返回。
+ 一旦进入文件列表或回放流程，会设置 `isWakeupVideoPlaybackMode = true`，后续查询更偏向真实设备流程，避免一直只拿旧缓存。

这个设计的目标是减少唤醒模式下频繁打开设备，同时又保证真正播放/下载前设备能被唤醒。

## 九、当时修改的整体思路

### 第一步：先解决回放连接不稳定

问题表现是播放后容易被停止、旧连接影响新连接。

处理思路：

+ 每次回放启动分配新的 session。
+ 旧 session 的 RTP 回调直接丢弃。
+ 停止回放时先清业务状态，再异步关闭 RTSP。
+ 避免底层关闭阻塞影响上层协议响应。

### 第二步：确认下载不是“文件上传”，而是“录像流推送”

根据服务器下发的 `9DH` 数据和日志，下载指令携带的仍然是服务器地址、端口、时间段和 SSRC。

所以设备端不需要新增“本地生成录像文件再上传”的流程，而是复用回放流：

+ 设备按时间段从机芯取录像。
+ 按服务器下发的 IP/端口推送。
+ 服务器侧负责接收并保存。

### 第三步：抽出 doPlaybackStream(...)

原来 `doPlayback(...)` 只服务 `9AH`。

为了让 `9DH` 复用解析逻辑，新增：

```java
private void doPlaybackStream(byte[] raw, byte responseOrder, String actionName)
```

其中：

+ `raw`：原始接收数据。
+ `responseOrder`：响应控制字，回放传 `ORDER_9AH`，下载传 `ORDER_9DH`。
+ `actionName`：日志显示名称，例如“启动回放”或“启动录像下载”。

这样可以保证：

+ 9AH 和 9DH 的 IP/端口/时间/SSRC 解析完全一致。
+ 后续协议字段如果调整，只需要改一个地方。
+ 日志仍然能区分回放和下载。

### 第四步：下载取消复用停止回放

因为下载和回放底层都是一条回放流，所以 `9EH` 不再只是原样回复，而是：

+ 解析通道和 SSRC。
+ 调用 `stopPlayCallBack(...)`。
+ 回复 `ORDER_9EH`。

这让后台下发“取消下载”时，设备端能真正停止正在推送的录像流。

## 十、当前代码需要注意的点

### 1. 当前 9DH 不是单独文件上传

当前实现中，“录像文件下载”不是把本地文件通过文件上传协议发出去，而是复用回放流推送。

如果后续服务器要求设备上传一个完整文件，需要重新设计：

+ 本地录像文件生成路径。
+ 文件分片。
+ 上传结束确认。
+ 失败重传。
+ 与当前 9DH 流式下载区分。

### 2. MainActivity 中的 SSRC 过期控制/停止过滤当前没有保留

历史提交 `0ae3af8` 曾在 `MainActivity` 中添加 `isStalePlaybackRequest(...)`，用于忽略 SSRC 不匹配的控制/停止指令。

但后续提交 `e13b6e5` 把这部分移除了。

当前仍然保留的稳定性措施主要在 `HuanYuDevice`：

+ `playbackSession` 过滤旧回调。
+ 停止时先清状态。
+ 异步执行 RTSP stop。

如果后续再次出现“服务器自动下发旧停止指令导致新回放被停”的问题，可以考虑恢复 `isStalePlaybackRequest(...)` 这类 SSRC 过滤。

### 3. 唤醒模式下要注意保活

录像查询、录像回放、录像下载都会触发设备上电和保活。

如果下载时间超过保活时间，但设备状态没有处于 `PLAYBACKING` 或其他业务态，可能会被空闲下电逻辑影响。

所以验证时要重点看：

+ 启动下载后设备是否进入回放/推流状态。
+ 保活到期时 `isWakeupActiveTask(...)` 是否能识别正在下载。
+ 下载取消后是否正常下电。

## 十一、建议验证日志关键字

验证回放：

+ `启动回放`
+ `回放启动成功 session=`
+ `回放断开`
+ `停止回放 session=`

验证下载：

+ `启动录像下载`
+ `回放启动成功 session=`
+ `录像下载取消`
+ `停止回放 session=`

验证唤醒模式：

+ `唤醒模式下播放红外录像文件`
+ `唤醒模式下播放可见光录像文件`
+ `唤醒模式下录像回放控制`
+ `唤醒模式下，保活时间已到`
+ `关闭云台`
+ `关闭红外`

## 十二、总结

这次修改的核心结论是：

+ 录像文件播放 `9AH` 和录像文件下载 `9DH` 在设备端本质上都是“按时间段回放录像并推送到服务器”。
+ 因此下载复用了回放流处理逻辑，只区分响应控制字和日志名称。
+ 下载取消 `9EH` 复用停止回放，保证能真正停止推流。
+ 回放稳定性主要通过 `HuanYuDevice.playbackSession`、SSRC 写入和异步停止 RTSP 来保证。
+ 唤醒模式下，录像查询、回放、下载都需要先上电并刷新保活，避免业务执行中被下电。
