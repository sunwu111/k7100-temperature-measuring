# 可见光机芯直播拉流与短视频录制并行设计方案

## 1. 存在的问题

当前测试问题：后台正在拉实时直播流时，如果再下发短视频录制命令，短视频可以录制成功，但直播会被打断。


## 2. 目标

本方案目标是让直播和短视频录制可以并行：

1. 直播过程中可以启动短视频录制。
2. 短视频录制不能主动停止直播。
3. 短视频录制结束后，只释放录制相关资源。
4. 直播资源从开始直播到收到停止直播命令前一直保持。


## 3. 当前逻辑问题

### 3.1 任务调度层的问题

当前短视频任务启动前，会先判断设备是否正在直播。

如果设备正在直播，会先停止直播，再进入短视频录制流程。

因此在调度层，直播和录制已经被定义成互斥关系。

### 3.2 设备资源层的问题

当前可见光机芯只有一套实时流客户端资源。

直播使用这套资源拉流，短视频录制也使用这套资源拉流。

当短视频录制启动时，会把原来的直播拉流连接关闭，然后重新建立一条用于录制的连接。

所以后台看到直播断流，本质上不是录制完成才断，而是录制启动时直播连接已经被替换掉了。

### 3.3 数据处理层的问题

当前直播收包逻辑同时承担两个职责：

1. 把实时 RTP 包转发给后台，形成直播。
2. 当设备处于录制状态时，把同一批 RTP 包写入 MP4。

这种设计适合“直播和录制共用同一条流”的模式，但不适合“两条流并行”的模式。

如果只是简单多开一条流，但仍然共用原来的收包逻辑，会出现录制流也被当成直播流转发、两条流互相争用解包状态等问题。

## 4. 并行处理的核心思路

并行处理的核心是把直播和短视频录制拆成两条独立链路。

### 4.1 直播链路

直播链路只负责实时直播：

1. 收到开始直播命令。
2. 打开 可见光 设备。
3. 建立直播拉流连接。
4. 收到 RTP 数据后，按直播协议上报给后台。
5. 直到收到停止直播命令，才释放直播连接和直播状态。

直播链路不负责写 MP4。

### 4.2 录制链路

录制链路只负责短视频文件：

1. 收到短视频录制命令。
2. 如果当前没有录制任务，则建立录制专用拉流连接。
3. 录制链路收到 RTP 数据后，只负责转换并写入 MP4。
4. 到达录制时长后，停止录制连接。
5. 关闭文件写入器。
6. 生成短视频文件并进入上传流程。

录制链路不负责向后台上报直播流。

### 4.3 状态关系

设计后，设备允许同时处于直播和录制状态。

直播状态表示当前主站正在看实时流。

录制状态表示当前正在生成短视频文件。

两者互不代替，也互不清除。

直播结束时，只清除直播状态。

录制结束时，只清除录制状态。

## 5. 推荐业务流程

### 5.1 直播中启动短视频

推荐流程如下：

1. 主站下发开始直播命令。
2. 设备建立直播链路，开始向主站上报实时流。
3. 主站下发短视频录制命令。
4. 系统发现设备正在直播，但不停止直播。
5. 系统检查当前是否已经在录制。
6. 如果没有录制，则启动录制链路。
7. 直播链路继续上报实时流。
8. 录制链路独立写 MP4。
9. 录制结束后，只释放录制链路。
10. 直播继续保持，直到主站明确下发停止直播命令。

### 5.2 重复录制

如果短视频录制过程中又收到新的短视频录制命令，继续沿用当前策略：不允许同一通道同时启动两个短视频录制任务。


### 5.3 直播停止

如果短视频录制过程中收到停止直播命令，系统只停止直播链路。


## 6. 唤醒模式下的处理

唤醒模式下，短视频录制完成后，原逻辑可能会根据后续任务情况决定是否下电。

设计后，需要增加一条保护规则：

如果短视频录制完成时设备仍在直播，则不能因为“短视频结束”触发下电。

原因是直播本身就是一个活跃业务，设备必须保持上电。

只有当直播已经停止、录制也结束，并且没有其他活跃任务时，才允许进入后续保活或下电判断。

## 7. 资源隔离原则

为了让直播和录制稳定并行，需要遵守以下资源隔离原则：

1. 直播使用直播专用拉流连接。
2. 录制使用录制专用拉流连接。
3. 直播使用直播专用 RTP 处理状态。
4. 录制使用录制专用 RTP 处理状态。
5. 直播停止不能释放录制连接。
6. 录制停止不能释放直播连接。
7. 录制数据不能进入直播上报通道。
